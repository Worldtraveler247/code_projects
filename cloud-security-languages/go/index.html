01Why Go for cloud security

You don't pick Go because it's the prettiest language. You pick it because the things you'll use every day in cloud infrastructure are written in it.

Kubernetes. Docker. Terraform. HashiCorp Vault. Consul. Prometheus. CoreDNS. Caddy. Nearly every modern observability and infrastructure tool that matters in the cloud-native ecosystem is Go. Reading their source — and writing extensions, controllers, or your own security tools that integrate with them — means reading and writing Go.

Concretely, what makes Go a good fit for security work

  • Single static binary. go build produces one file you can scp to a server, drop in a container, or send to an air-gapped host. No runtime, no virtualenv, no pip install on the target.
  • Cross-compilation built in. GOOS=linux GOARCH=arm64 go build from your Mac. Useful when you're building tools that run on whatever odd box the SOC needs them on.
  • Concurrency that fits scanning, polling, and event handling. Goroutines and channels are first-class. A 200-target port scanner is ~50 lines.
  • Strong, fast standard library. net/http, crypto/tls, encoding/json, os/exec — you rarely need third-party deps for security tooling.
  • Static typing without ceremony. The compiler catches whole classes of bugs before they ship — the kind of bugs that turn into CVEs in long-lived security tools.
Honest tradeoff

Go is verbose where Python is concise. if err != nil { return err } appears thousands of times in any real Go codebase. You'll read more lines, but you'll also debug fewer surprise crashes at 3am. For long-running infrastructure code that's the right trade. For a one-off scrape-and-parse script, Python wins.

02Setup — Go + VS Code

Install Go

On macOS, the easiest path:

terminal
brew install go
go version
# go version go1.22.x darwin/arm64

Or download the official installer from go.dev/dl. Either works; Homebrew makes upgrades easier.

Verify your environment

terminal
go env GOPATH GOROOT GOOS GOARCH
# /Users/you/go    /opt/homebrew/Cellar/go/1.22.x/libexec    darwin    arm64

GOPATH is where modules and binaries live by default ($HOME/go). GOROOT is where Go itself is installed. You almost never need to change either.

Set up VS Code

  1. Install VS Code from code.visualstudio.com.
  2. Open the Extensions panel (Cmd+Shift+X) and install Go by Google (publisher: golang.go).
  3. Open any .go file. VS Code will prompt to install Go tools (gopls, dlv, etc.). Click Install All.
  4. You now have: autocomplete, jump-to-definition, error highlighting, formatting on save, and an integrated debugger.

Reference: VS Code Go extension docs.

Your first program

Create a folder, then a file:

terminal
mkdir -p ~/go-learning/hello && cd ~/go-learning/hello
go mod init hello
touch main.go
code .   # opens VS Code in this folder

In main.go:

main.go
package main

import "fmt"

func main() {
    fmt.Println("Hello, Go.")
}

Run it:

terminal
go run main.go
# Hello, Go.

go build -o hello main.go
./hello
# Hello, Go.

go run compiles and runs in one step (good for development). go build produces a binary you can ship. Reference: go.dev/doc/tutorial/getting-started.

03Anatomy of a Go program

Every Go file has the same skeleton. Let's read the smallest possible program one line at a time.

main.go
package main

import "fmt"

func main() {
    fmt.Println("Hello, Go.")
}

package main

Every Go file starts with a package declaration. A package is Go's unit of code organization — like a Python module or a Java package. The name main is special: it tells the Go toolchain "this is a runnable program, not a library." Every other package name produces a library that other Go code can import.

Reference: Go spec — Packages.

import "fmt"

Brings in the fmt package from the standard library. fmt handles formatted I/O — printing, scanning, formatting strings. After importing, you reference its exported names via fmt.Println, fmt.Printf, etc.

For multiple imports, use parentheses:

go
import (
    "fmt"
    "os"
    "strings"
)

Reference: pkg.go.dev/fmt.

func main() { ... }

Declares a function called main. In package main, this function is the program's entry point — Go runs it when the binary starts. It takes no arguments and returns nothing.

Two non-negotiable Go style rules the compiler / formatter enforce:

  • The opening { goes on the same line as the function declaration. Putting it on the next line is a syntax error.
  • Indentation is tabs, formatted by gofmt. The VS Code Go extension runs gofmt on save automatically. Don't fight it.

fmt.Println("Hello, Go.")

Calls the Println function from the fmt package with the string "Hello, Go." as its argument. Println writes its arguments to standard output followed by a newline.

Capitalization matters

In Go, a name is exported (visible outside its package) if and only if it starts with a capital letter. fmt.Println is exported; fmt.println would not exist outside the fmt package. This is the entire access-control system. No public, private, or protected keywords.

04Packages and imports

A Go module is a collection of related packages, identified by a module path (typically a repo URL). The go.mod file at the root of a module declares it.

terminal
go mod init github.com/worldtraveler247/portscan
# creates go.mod with: module github.com/worldtraveler247/portscan

Standard library imports

go
import (
    "fmt"      // formatted I/O
    "os"       // operating system interface
    "net"      // TCP/UDP/HTTP networking
    "strings"  // string utilities
)

Third-party imports

You add a dependency by importing it and running go mod tidy. Go fetches the module, records the version in go.mod, and locks checksums in go.sum.

go
import "github.com/spf13/cobra"
terminal
go mod tidy
# go: downloading github.com/spf13/cobra v1.8.0
# go: added github.com/spf13/cobra v1.8.0

Import aliases and the blank identifier

go
import (
    j "encoding/json"   // alias: use as j.Marshal(...)
    _ "net/http/pprof"  // import for side effects only (registers handlers)
)

References: Go Modules reference, Managing dependencies.

05Variables

Go has several ways to declare a variable. Use the shortest one that works in context.

Full form: var name type = value

go
var name string = "Eddie"
var age  int    = 30
var on   bool   = true

Type inferred: var name = value

go
var name = "Eddie"   // string, inferred from "Eddie"
var age  = 30        // int, inferred from 30
var pi   = 3.14159   // float64, inferred

Short declaration: name := value (inside functions only)

go
func main() {
    name := "Eddie"
    age  := 30
    fmt.Println(name, age)
}

Multiple variables in one line

go
var x, y int = 1, 2
a, b := "hello", 42       // different types are fine

// Common pattern: function returning multiple values
host, port := parseAddr(":8080")

Zero values

Every variable in Go is initialized. If you don't give it a value, Go gives it the type's zero value:

TypeZero value
int, float64, all numeric0
string"" (empty string)
boolfalse
pointers, slices, maps, channels, functions, interfacesnil
structseach field gets its own zero value

06Constants

Constants are values fixed at compile time. They cannot be modified at runtime and they cannot be declared with :=.

go
const Pi = 3.14159
const GreetingPrefix = "Hello, "
const MaxRetries = 5

// Multiple constants
const (
    KB = 1024
    MB = 1024 * KB
    GB = 1024 * MB
)

iota — auto-incrementing constant generator

iota is Go's enum-builder. Inside a const ( ... ) block, it starts at 0 and increments by 1 for each line.

go
const (
    LogDebug = iota   // 0
    LogInfo           // 1
    LogWarn           // 2
    LogError          // 3
)

// Bitmask flags
const (
    PermRead    = 1 << iota   // 1
    PermWrite                  // 2
    PermExecute                // 4
)

References: Tour: Constants, Go spec — iota.

08Conditionals

if / else if / else

go
if port < 1024 {
    fmt.Println("privileged port")
} else if port < 49152 {
    fmt.Println("registered port")
} else {
    fmt.Println("ephemeral port")
}

No parentheses around the condition (the compiler will reject them). Braces are required even for one-line bodies.

if with an initialization statement

This is the single most common Go pattern. You'll write it a thousand times.

go
if conn, err := net.Dial("tcp", "1.1.1.1:80"); err != nil {
    log.Fatal(err)
} else {
    defer conn.Close()
    // use conn
}

conn and err are scoped to the if/else block. Any variables you declare in the init statement vanish after.

switch

Go's switch is more powerful than C's and has no implicit fallthrough.

go
switch os := runtime.GOOS; os {
case "darwin":
    fmt.Println("macOS")
case "linux":
    fmt.Println("Linux")
case "windows":
    fmt.Println("Windows")
default:
    fmt.Printf("%s\n", os)
}

// Switch with no expression = clean if/else if chain
switch {
case statusCode >= 500:
    fmt.Println("server error")
case statusCode >= 400:
    fmt.Println("client error")
case statusCode >= 300:
    fmt.Println("redirect")
default:
    fmt.Println("ok")
}

References: Tour: If, Tour: Switch.

09Loops

Go has exactly one looping keyword: for. It covers every iteration shape other languages spread across for, while, do/while, and foreach.

C-style three-clause

go
for i := 0; i < 5; i++ {
    fmt.Println(i)
}

While-style (condition only)

go
n := 1
for n < 1000 {
    n *= 2
}
fmt.Println(n)   // 1024

Infinite loop

go
for {
    if shouldStop() {
        break
    }
    work()
}

for ... range — iterate over collections

go
// Slice
ports := []int{22, 80, 443, 8080}
for index, port := range ports {
    fmt.Printf("[%d] %d\n", index, port)
}

// Map (order is randomized intentionally)
headers := map[string]string{
    "X-Forwarded-For": "10.0.0.1",
    "User-Agent":      "scanner/1.0",
}
for key, value := range headers {
    fmt.Printf("%s: %s\n", key, value)
}

// String — iterates over Unicode code points (runes), not bytes
for i, r := range "héllo" {
    fmt.Printf("%d %c\n", i, r)
}

break, continue, labels

go
outer:
for i := 0; i < 10; i++ {
    for j := 0; j < 10; j++ {
        if i*j > 50 {
            break outer   // breaks the labeled loop, not just the inner one
        }
    }
}

Reference: Tour: For.

10Functions

Basic shape

go
func add(a int, b int) int {
    return a + b
}

// Same-type parameters can share a type declaration
func add2(a, b int) int {
    return a + b
}

Multiple return values

The defining feature of Go function signatures. Used pervasively for "value, error" returns.

go
func divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, fmt.Errorf("division by zero")
    }
    return a / b, nil
}

result, err := divide(10, 2)
if err != nil {
    log.Fatal(err)
}
fmt.Println(result)   // 5

Named returns

go
func divmod(a, b int) (quotient, remainder int) {
    quotient = a / b
    remainder = a % b
    return    // "naked return" — returns the named values
}

Useful for short functions where the names document intent. For long functions, prefer explicit returns — the indirection becomes confusing.

Variadic functions

go
func sum(nums ...int) int {
    total := 0
    for _, n := range nums {
        total += n
    }
    return total
}

sum(1, 2, 3)              // 6
nums := []int{1, 2, 3, 4}
sum(nums...)              // spread the slice into individual args

Anonymous functions and closures

go
// Anonymous function called immediately
func() {
    fmt.Println("inline")
}()

// Closure: captures its enclosing scope
func makeCounter() func() int {
    count := 0
    return func() int {
        count++
        return count
    }
}

next := makeCounter()
fmt.Println(next(), next(), next())   // 1 2 3

Functions are values

go
operations := map[string]func(int, int) int{
    "add": func(a, b int) int { return a + b },
    "mul": func(a, b int) int { return a * b },
}
fmt.Println(operations["add"](3, 4))   // 7

Reference: Tour: Functions.

11Composite types — arrays, slices, maps

Arrays — fixed size

go
var ports [3]int = [3]int{22, 80, 443}
fmt.Println(ports[0])     // 22
fmt.Println(len(ports))   // 3

You'll rarely use arrays directly. Slices are what you want.

Slices — dynamic size, the workhorse

go
var ports []int                // nil slice, len 0
ports = []int{22, 80, 443}     // literal
ports = append(ports, 8080)    // grow
fmt.Println(len(ports), cap(ports))   // 4 4

// make(): allocate with known length and capacity
buf := make([]byte, 1024)            // len=1024, cap=1024
buf := make([]byte, 0, 1024)         // len=0, cap=1024 — grow without reallocation

// Slicing a slice
fmt.Println(ports[1:3])              // [80 443]
fmt.Println(ports[:2])               // [22 80]
fmt.Println(ports[2:])               // [443 8080]
Slice gotcha

Slices share an underlying array. Modifying a slice modifies any other slice that overlaps the same backing array. If you need an independent copy, use copy() or append([]T{}, src...).

Maps — key/value pairs

go
// Literal
ports := map[string]int{
    "ssh":   22,
    "http":  80,
    "https": 443,
}

// make()
counts := make(map[string]int)
counts["denied"]++   // works on zero value (0)

// Lookup with comma-ok (existence check)
if port, ok := ports["smtp"]; ok {
    fmt.Println("smtp on", port)
} else {
    fmt.Println("smtp not configured")
}

// Delete
delete(ports, "http")

References: Tour: Slices, Go blog: Slices.

12Pointers

A pointer holds the memory address of a value. Go has pointers, but no pointer arithmetic — much safer than C.

go
x := 42
p := &x          // & takes the address — p is *int
fmt.Println(*p)   // * dereferences — prints 42

*p = 100         // assign through the pointer
fmt.Println(x)    // 100

When to use pointers

  • Mutation. If a function needs to modify its argument, pass a pointer.
  • Avoiding copies of large structs. Passing a pointer is one word; passing a struct by value copies every field.
  • Signaling absence. A *string can be nil; a string cannot.

Struct field access through a pointer auto-dereferences

go
type Server struct{ Port int }
s := &Server{Port: 8080}
fmt.Println(s.Port)   // not (*s).Port — Go does this for you

Reference: Tour: Pointers.

13Structs and methods

Defining a struct

go
type ScanResult struct {
    Host    string
    Port    int
    Open    bool
    Banner  string
}

Creating instances

go
// Named fields (preferred — explicit)
r1 := ScanResult{
    Host: "1.1.1.1",
    Port: 443,
    Open: true,
}

// Positional (fragile — breaks if field order changes)
r2 := ScanResult{"1.1.1.1", 443, true, ""}

// Zero value of all fields
var r3 ScanResult
fmt.Printf("%+v\n", r3)
// {Host: Port:0 Open:false Banner:}

Methods

A method is a function with a receiver. The receiver argument appears between func and the method name.

go
func (r ScanResult) String() string {
    state := "closed"
    if r.Open {
        state = "open"
    }
    return fmt.Sprintf("%s:%d %s", r.Host, r.Port, state)
}

r := ScanResult{Host: "1.1.1.1", Port: 443, Open: true}
fmt.Println(r.String())   // 1.1.1.1:443 open

Value vs pointer receivers

go
// Value receiver: the method gets a copy. Cannot mutate r.
func (r ScanResult) Summary() string { ... }

// Pointer receiver: the method can mutate r.
func (r *ScanResult) MarkOpen() {
    r.Open = true
}

Convention: if any method on a type uses a pointer receiver, use pointer receivers for all methods on that type. Mixing is allowed but almost always a smell.

Struct embedding (composition)

Go has no inheritance. It has embedding, which is more honest.

go
type Logger struct{ Prefix string }
func (l Logger) Log(msg string) { fmt.Printf("[%s] %s\n", l.Prefix, msg) }

type Scanner struct {
    Logger      // embedded; no field name
    Targets []string
}

s := Scanner{Logger: Logger{Prefix: "scan"}, Targets: []string{"1.1.1.1"}}
s.Log("starting")   // promoted method call — same as s.Logger.Log("starting")
// [scan] starting

Reference: Tour: Structs.

14Interfaces

An interface is a set of method signatures. Any type that has all those methods satisfies the interface. There's no implements keyword — satisfaction is implicit, structural, checked at compile time.

Defining and satisfying an interface

go
type Stringer interface {
    String() string
}

// ScanResult already has String() string from the previous section,
// so it automatically satisfies Stringer. No declaration needed.

func describe(s Stringer) {
    fmt.Println(s.String())
}

describe(ScanResult{Host: "1.1.1.1", Port: 443, Open: true})

The empty interface and any

interface{} has no methods, so every type satisfies it. Since Go 1.18, any is a built-in alias for it. Use sparingly — you lose type safety.

go
func printAnything(v any) {
    fmt.Printf("%v (%T)\n", v, v)
}

printAnything(42)         // 42 (int)
printAnything("hello")    // hello (string)

Type assertions

go
var i any = "hello"

s := i.(string)         // panics if i is not a string
s, ok := i.(string)     // safe form: ok is false if assertion fails

if n, ok := i.(int); ok {
    fmt.Println("got int:", n)
} else {
    fmt.Println("not an int")
}

Type switches

go
func describe(i any) string {
    switch v := i.(type) {
    case int:
        return fmt.Sprintf("int: %d", v)
    case string:
        return fmt.Sprintf("string: %q", v)
    case error:
        return fmt.Sprintf("error: %v", v)
    default:
        return fmt.Sprintf("unknown: %T", v)
    }
}

Standard-library interfaces worth knowing on day one

InterfaceMethodWhy it matters
errorError() stringEvery error in Go satisfies this
fmt.StringerString() stringCustomizes how %v prints your type
io.ReaderRead(p []byte) (n int, err error)Any source of bytes — files, network, strings
io.WriterWrite(p []byte) (n int, err error)Any sink for bytes

References: Tour: Interfaces, Effective Go: Interfaces.

15Errors

Go's most distinctive design choice: errors are values, not exceptions. Every function that can fail returns an error as its last value, and you check it.

The error interface

go
type error interface {
    Error() string
}

Built into the language. Anything with an Error() string method is an error.

The pattern you will write thousands of times

go
data, err := os.ReadFile("/etc/hosts")
if err != nil {
    return err
}
// ... use data

Creating errors

go
import "errors"

err := errors.New("scan timed out")
err := fmt.Errorf("connect to %s:%d failed: %w", host, port, cause)
//                                                ^^ wraps the underlying error

Inspecting wrapped errors

go
import "errors"

var ErrTimeout = errors.New("scan timed out")

err := scan("1.1.1.1")
if errors.Is(err, ErrTimeout) {
    // err or anything it wraps == ErrTimeout
    retryLater()
}

var pathErr *os.PathError
if errors.As(err, &pathErr) {
    // err or anything it wraps is a *os.PathError
    fmt.Println("path involved:", pathErr.Path)
}

Custom error types

go
type ScanError struct {
    Host string
    Port int
    Err  error
}

func (e *ScanError) Error() string {
    return fmt.Sprintf("scan %s:%d: %v", e.Host, e.Port, e.Err)
}
func (e *ScanError) Unwrap() error { return e.Err }   // makes errors.Is/As work

panic and recover (use sparingly)

go
// panic — for genuinely unrecoverable situations.
// In normal control flow, return an error instead.

func mustHave(cond bool, msg string) {
    if !cond {
        panic(msg)
    }
}

// recover — turns a panic into an error you can handle.
// Only useful inside a deferred function.
func safeCall(fn func()) (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("recovered: %v", r)
        }
    }()
    fn()
    return nil
}
The Go philosophy

"Don't just check errors, handle them gracefully." Errors carry context. Wrap them as they bubble up so the final log line tells the full story: "scan: connect tcp 1.1.1.1:443: i/o timeout" rather than just "timeout".

References: Go blog: Error handling and Go, Working with errors in Go 1.13+.

16A taste of concurrency

This is a deep topic — entire books are written about it. Here's just enough to recognize it when you see it.

Goroutines

Prefix any function call with go and it runs concurrently.

go
go fetch("https://example.com")
go fetch("https://example.org")
// main keeps going immediately; both fetches run in the background

Channels

Goroutines coordinate by sending values through channels.

go
results := make(chan string)

go func() {
    results <- "hello from goroutine"   // send
}()

msg := <-results                          // receive (blocks until a send happens)
fmt.Println(msg)

The Go concurrency mantra

From the Go proverbs

"Don't communicate by sharing memory; share memory by communicating." Use channels to pass values between goroutines instead of using mutexes around shared state. Mutexes are available (sync.Mutex) but channels are the idiomatic default.

References: Tour: Concurrency, Effective Go: Concurrency.

func main() {

Drill 3 — Reverse a string

drill3.go
package main

import "fmt"

func reverse(s string) string {
    runes := []rune(s)
    for i, j := 0, len(runes)-1; i < j; i, j = i+1, j-1 {
        runes[i], runes[j] = runes[j], runes[i]
    }
    return string(runes)
}

func main() {
    fmt.Println(reverse("cloud security"))   // ytiruces duolc
}

Drill 4 — Sum of command-line numbers

drill4.go
package main

import (
    "fmt"
    "os"
    "strconv"
)

func main() {
    sum := 0
    for _, arg := range os.Args[1:] {
        n, err := strconv.Atoi(arg)
        if err != nil {
            fmt.Fprintf(os.Stderr, "skipping %q: %v\n", arg, err)
            continue
        }
        sum += n
    }
    fmt.Println("sum:", sum)
}
// Run: go run drill4.go 1 2 3 4 5     -> sum: 15

Drill 5 — Count lines in a file

drill5.go
package main

import (
    "bufio"
    "fmt"
    "os"
)

func main() {
    if len(os.Args) < 2 {
        fmt.Fprintln(os.Stderr, "usage: drill5 <file>")
        os.Exit(1)
    }
    f, err := os.Open(os.Args[1])
    if err != nil {
        fmt.Fprintln(os.Stderr, err)
        os.Exit(1)
    }
    defer f.Close()

    count := 0
    sc := bufio.NewScanner(f)
    for sc.Scan() {
        count++
    }
    fmt.Println("lines:", count)
}

Drill 6 — Word frequency

drill6.go
package main

import (
    "bufio"
    "fmt"
    "os"
    "strings"
)

func main() {
    counts := make(map[string]int)
    sc := bufio.NewScanner(os.Stdin)
    for sc.Scan() {
        for _, word := range strings.Fields(sc.Text()) {
            counts[strings.ToLower(word)]++
        }
    }
    for word, n := range counts {
        fmt.Printf("%4d %s\n", n, word)
    }
}

Drill 9 — Concurrent URL pinger

drill9.go
package main

import (
    "fmt"
    "net/http"
    "sync"
    "time"
)

func ping(url string, wg *sync.WaitGroup) {
    defer wg.Done()
    start := time.Now()
    resp, err := http.Get(url)
    if err != nil {
        fmt.Printf("%-30s ERROR %v\n", url, err)
        return
    }
    resp.Body.Close()
    fmt.Printf("%-30s %s in %v\n", url, resp.Status, time.Since(start))
}

func main() {
    urls := []string{
        "https://go.dev",
        "https://kubernetes.io",
        "https://aws.amazon.com",
    }
    var wg sync.WaitGroup
    for _, u := range urls {
        wg.Add(1)
        go ping(u, &wg)
    }
    wg.Wait()
}

Drill 10 — Concurrent TCP port scanner

drill10.go
package main

import (
    "fmt"
    "net"
    "sync"
    "time"
)

func scan(host string, port int, wg *sync.WaitGroup) {
    defer wg.Done()
    addr := fmt.Sprintf("%s:%d", host, port)
    conn, err := net.DialTimeout("tcp", addr, 1*time.Second)
    if err != nil {
        return
    }
    conn.Close()
    fmt.Printf("%s open\n", addr)
}

func main() {
    host := "scanme.nmap.org"   // explicitly authorizes scanning
    var wg sync.WaitGroup
    for port := 1; port <= 1024; port++ {
        wg.Add(1)
        go scan(host, port, &wg)
    }
    wg.Wait()
}
Only scan what you own or are authorized to scan

scanme.nmap.org is run by the Nmap project specifically to be a legal scan target. Pointing port scanners at random hosts on the internet is computer-crime-statute territory in most jurisdictions. For practice, run a vulnerable container locally (e.g., DVWA, OWASP Juice Shop) and scan that.

1890-day learning plan (honest)

First, a reality check on "expert"

No one becomes a Go expert in 90 days. Real expertise — knowing the runtime, reading the source of net/http, designing production-grade concurrent systems, mentoring others — takes 2–5 years of regular use. What you can absolutely reach in 90 days is competent: comfortable enough to read most Go code, write small-to-medium tools, and ship something that handles real input safely. That's the goal here.

Days 1–7 — Syntax fluency

Days 1–2

Install Go and VS Code. Read sections 1–7 of this lesson. Type drills 1–3 from scratch (don't copy-paste).

~2 hours/day

Days 3–5

Read sections 8–12. Type drills 4–6. Run them, break them on purpose, read the error messages until the messages stop being scary.

~2 hours/day

Days 6–7

Read sections 13–15. Type drills 7–8. Start the official Tour of Go end-to-end on the side.

~3 hours over weekend

Days 8–21 — Standard library comfort

Week 2

Deep dive: fmt, strings, strconv, os, bufio, errors. Build a CLI tool that takes a CSV file and produces a sorted summary.

~1 hour/day weekday, 3 hours/weekend day

Week 3

Deep dive: encoding/json, net/http, io. Build a CLI that takes a URL, fetches it, parses JSON, and pretty-prints a summary.

~1 hour/day

Days 22–45 — Real programs

Weeks 4–5

Learn flag for CLI argument parsing. Rebuild your CSV tool with proper flags and subcommands. Add unit tests using the built-in testing package.

Week 6

Build a small HTTP server using net/http: 3 endpoints, JSON responses, structured logging with log/slog. Containerize it with a Dockerfile.

Days 46–66 — Concurrency for real

Weeks 7–8

Goroutines, channels, sync.Mutex, sync.WaitGroup, context.Context. Read Go Concurrency Patterns: Pipelines.

Week 9

Rewrite drill 10 (port scanner) with bounded concurrency (worker pool) and per-target timeouts using context.WithTimeout.

Days 67–90 — Ship something real

Weeks 10–13

Pick a project that solves an actual problem you have. Suggestions for a cloud security path: an AWS IAM policy diff tool, a TLS cert expiration checker for a list of domains, a CloudTrail event filter, a small file-integrity monitor for a directory tree.

Quality bar

Tests for the core logic. Proper error handling (no panic in normal flow). A README that someone else could actually use. Cross-compiled binaries for at least linux/amd64 and darwin/arm64.

Beyond day 90

You're competent. To get to expert, the path is reading and writing a lot more Go: contribute to an open-source Go project, read parts of the standard library source (net/http is famously approachable), and read The Go Programming Language cover to cover. Plan for another year or two of regular practice.

19Books and audio for advanced study

The books below are the ones experienced Go engineers actually recommend when asked. Order them by where you are: the beginner books first, the domain-specific ones once you've shipped something real.

The five books that come up in every "what should I read" thread

  • The Go Programming Language — Alan A. A. Donovan & Brian W. Kernighan (Addison-Wesley, 2015). Universally called "GOPL" or "the K&R of Go." Kernighan co-wrote The C Programming Language; the same precision and economy. Slightly dated on tooling and modules, but the language fundamentals haven't changed and no other book treats them with the same rigor. Buy this around month 2–3. Publisher: gopl.io.
  • Learning Go: An Idiomatic Approach to Real-World Go Programming — Jon Bodner (O'Reilly, 2nd ed. 2024). The current best beginner-to-intermediate book. Covers Go 1.21+, generics, modules, structured logging, the modern toolchain. Reading this alongside the Tour gives you a complete first pass. Publisher: O'Reilly catalog.
  • 100 Go Mistakes and How to Avoid Them — Teiva Harsanyi (Manning, 2022). Anti-patterns and their fixes. Reads like a senior engineer code-reviewing your work in advance. The right book to bridge "I can write Go" → "I write Go that won't embarrass me." Read at month 4–6. Publisher: Manning.
  • Concurrency in Go: Tools and Techniques for Developers — Katherine Cox-Buday (O'Reilly, 2017). The definitive treatment of goroutines, channels, the sync package, and pipeline patterns. Slightly dated (predates context being everywhere) but the underlying model is unchanged. Pair with the official Go Concurrency Patterns blog posts. Publisher: O'Reilly catalog.
  • Cloud Native Go: Building Reliable Services in Unreliable Environments — Matthew A. Titmus (O'Reilly, 2nd ed. 2024). Direct match for your trajectory. Covers the patterns Go is actually used for in cloud infrastructure: idempotency, circuit breakers, observability, deployment, the SRE-adjacent design choices. Read this in parallel with whatever cloud project you build at the end of the 90-day plan. Publisher: O'Reilly catalog.

Specialty books for the cloud-security path

  • Black Hat Go: Go Programming for Hackers and Pentesters — Tom Steele, Chris Patten, Dan Kottmann (No Starch Press, 2020). Building actual security tools in Go: TCP scanners, packet manipulation, HTTP fuzzers, exploit delivery, C2 framework basics. Treat the offensive examples as blue-team understanding — read it to know what an attacker might build, not to point it at systems you don't own. Publisher: No Starch Press.
  • Network Programming with Go — Adam Woodbeck (No Starch Press, 2021). TCP, UDP, TLS, HTTP/2, streaming, custom protocols — all from the standard library, no third-party dependencies. The practical companion to Cloud Native Go. Essential if you'll write security tooling. Publisher: No Starch Press.
  • Distributed Services with Go — Travis Jeffery (Pragmatic Bookshelf, 2021). Build a distributed log service from scratch — gRPC, Raft consensus, service discovery, deployment to Kubernetes. Useful for understanding how the systems you'll secure are actually built. Publisher: Pragmatic Bookshelf.
  • Powerful Command-Line Applications in Go — Ricardo Gerardi (Pragmatic Bookshelf, 2021). Building production-quality CLIs with cobra, viper, testing, releasing. Most security tools you'll write are CLIs; this is the craftsmanship book. Publisher: Pragmatic Bookshelf.
  • The Go Programming Language Specification — The Go Authors (free, online). Not a tutorial — the actual normative document defining the language. Surprisingly readable. Read once at month 6+ when you can appreciate the precision. Source: go.dev/ref/spec.

Audio — an honest take

Programming books don't really work as audiobooks

None of the books above have audiobook versions, and that's correct: a narrator reading for i := 0; i < len(slice); i++ { aloud is useless. The few "Go" titles on Audible are usually low-effort summaries of free content. Save your Audible credits.

For ear-based learning, podcasts and conference talks are what actually works — they're conversations about Go rather than reading code verbatim. Recommended:

  • Go Time (Changelog Media) — weekly podcast with Go core-team members and prominent community engineers. The single best Go podcast. Conversational, technical, no fluff. Start with episodes on topics you care about (concurrency, generics, observability) rather than chronologically. Listen: changelog.com/gotime.
  • Cup o' Go — biweekly news podcast covering the Go ecosystem. Shorter, news-focused. Good for keeping up with releases, proposals, and community happenings without reading the mailing lists. Listen: cupogo.dev.
  • GopherCon talks on YouTube — the annual Go conference, archived. Audio works most of the time; glance at the slides when needed. Search for talks by Bill Kennedy (concurrency), Rob Pike (language design), Dave Cheney (performance), Mat Ryer (HTTP services), Robert Griesemer (generics). Channel: YouTube — GopherAcademy.
  • Just for Func — Francesc Campoy's series of short Go videos. Older (2016–2019) but still excellent for fundamentals. Each video is 10–20 minutes on one focused topic. Channel: YouTube — Just for Func.
  • Ardan Labs — Bill Kennedy talks Bill Kennedy (Ardan Labs) is one of the most respected Go educators. His talks on the Go memory model, scheduler, and idiomatic concurrency are foundational. Many are free on YouTube. Search: YouTube — Bill Kennedy / Ardan Labs.
  • The Changelog (general, not Go-only) — long-form interviews with notable open-source maintainers, including frequent Go ecosystem episodes. Good context for who's who in the cloud-native world. Listen: changelog.com/podcast.
