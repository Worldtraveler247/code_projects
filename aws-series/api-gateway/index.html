Amazon API Gateway

The bouncer at the door of your API. It checks IDs (auth), enforces rate limits, transforms requests, and only routes the good ones to your backend. Lambda's best friend and the most-used front door in serverless.

01 What is API Gateway?

A fully managed service that lets you create, publish, secure, and monitor APIs at any scale. Clients hit your gateway URL; the gateway authenticates them, throttles them, transforms their requests if needed, and forwards the survivors to a backend (Lambda, EC2/ALB, a private VPC service, or even another HTTP endpoint).

Three flavors — pick the right one

Type What it's for Price Notes
HTTP API New REST APIs in front of Lambda or HTTP backends $1.00 / 1M calls V2. Default choice for new work. Cheaper, lower latency, JWT auth built in.
REST API Mature feature set, private APIs, request/response transformation, caching, API keys + usage plans $3.50 / 1M calls V1. Use when you need features HTTP API doesn't support yet.
WebSocket API Real-time bi-directional connections (chat, live dashboards, multiplayer state) $1.00 / 1M msgs + $0.25 / 1M conn-min Stateful. Backend is usually Lambda for connect / disconnect / message handlers.

What the gateway does for you (so your backend doesn't have to)

  • Authentication — Cognito user pools, JWT verification, Lambda authorizers, IAM-signed requests, mutual TLS for B2B.
  • Rate limiting — per-stage, per-method, or per-API-key. Stops a misbehaving client from melting your backend.
  • Request validation — JSON Schema validation rejects malformed requests before they cost you Lambda invocations.
  • Request / response transformation — rewrite headers, body, status codes. Useful for legacy backend façades.
  • Caching (REST API) — TTL-based response caching. Huge savings for read-heavy APIs.
  • Custom domains + ACM certs — api.yourdomain.com, valid TLS, no work.
  • WAF integration — bolt on AWS WAF for OWASP rule sets, IP reputation, bot blocking.
  • Logging + tracing — full request logs to CloudWatch, X-Ray distributed traces, metrics by route.
In one sentence

API Gateway is the bouncer at the door of your API — it does auth, rate limiting, validation, and routing as configuration, so your backend code can focus on business logic.

02 A vibrant example: launch day for a weather app

Imagine you've built a weather app. Day 1 you have 10 users and a Flask app on a single EC2 instance. Fine. Then you get featured on the App Store. Now you have 10 million users hammering /forecast?lat=...&lon=... between 7 and 9 a.m. local time, every time zone in the world.

Without API Gateway: you're manually scaling EC2 instances, configuring an ALB, writing rate-limit middleware, debugging connection-pool exhaustion, fighting the noisy neighbor on a shared server, and getting paged. You'll spend the next two weeks doing infrastructure work instead of building your app.

With API Gateway + Lambda: you wrote a 30-line Lambda function. You configured a route in API Gateway. AWS handled everything else — TLS, certificate renewal, scaling from 0 to 10 million RPS, throttling, logging. The bill goes up; the latency doesn't move; you keep building features.

That's the pitch. You stop running infrastructure and start writing handlers. The cost is some lock-in (Lambda has limits, API Gateway has its own quirks) and some learning curve. For a huge class of apps, it's the right tradeoff.

03 Who uses API Gateway — and why

Almost any team building a public or internal HTTP API on AWS. Concrete examples (publicly documented):

🏦 Capital One

Has migrated huge swaths of their public API surface to API Gateway + Lambda. AWS case study →

🛡️ Liberty Mutual

Insurance quote engines on serverless — API Gateway as the entry point for hundreds of microservices.

📰 The Guardian

UK news outlet uses API Gateway for content delivery APIs and webhook receivers from publishing tools.

🤖 iRobot

Roomba telemetry and control APIs — tens of millions of devices reporting in. WebSocket API for real-time control.

🎮 Indie game studios

Match-making, leaderboards, player state — WebSocket APIs let small teams ship multiplayer features without running game servers.

🪝 Webhook receivers

GitHub / Stripe / Twilio webhooks land at an API Gateway endpoint, get verified by a Lambda authorizer, then trigger downstream work. The 80%-use-case.

Common reasons to reach for API Gateway

  • Serverless apps — the canonical Lambda pairing. No server, no patching, scales to zero.
  • Public APIs — auth, throttling, monetization (API keys + usage plans), per-customer rate limits.
  • Microservice façade — one stable URL in front of many internal services. Hide implementation details.
  • Webhook ingress — third-party services (Stripe, GitHub, Twilio) need a stable HTTPS endpoint with valid TLS.
  • Real-time apps — WebSocket APIs for chat, live scores, multiplayer game state.
  • Mobile / IoT backends — devices speak HTTP; API Gateway absorbs the spikes.

When NOT to use API Gateway

  • High-throughput sustained traffic where ALB + ECS/EKS is dramatically cheaper at scale (think 100k+ RPS sustained).
  • Latency-critical APIs — API Gateway adds 10–30ms overhead. Sub-10ms targets need ALB or NLB.
  • gRPC — API Gateway doesn't support it. Use ALB or AppMesh.
  • Heavy request-body transformation — gets unwieldy in Velocity templates. Often easier in code.
  • Anything you can't put behind 30-second timeouts — API Gateway has hard limits (29s integration timeout for REST/HTTP).

04 Competitors & positioning

The "API gateway" pattern is widely recognized — every major cloud and several open-source projects implement it. Honest framing: AWS API Gateway is the easiest path on AWS, but it's not the cheapest at scale, not the fastest, and not the most feature-rich.

Service Type Where it wins / loses
AWS API Gateway Managed, AWS-only Deepest AWS integration (Lambda, IAM, Cognito, WAF). Loses on: price at very high volumes, cold-start latency on REST API, configuration complexity for edge cases.
Kong / Kong Gateway Open source + enterprise Self-host anywhere (incl. on-prem), huge plugin ecosystem, Lua scripting. Multi-cloud. Loses on: you operate it.
Google Apigee Managed, enterprise Best analytics + developer portal in the category. Strong API monetization features. Loses on: price, complexity for simple use cases.
Azure API Management Managed Tight Active Directory + Office 365 integration. Built-in developer portal. Loses on: price tiers, regional model.
Cloudflare API Gateway Edge-native Runs at Cloudflare's 300+ edge locations — global low latency. Bundles WAF + DDoS. Loses on: AWS-native integration, fewer mature features.
NGINX / Traefik / HAProxy Self-hosted reverse proxy Very fast, very cheap if you have ops talent. Loses on: you build the auth, rate limiting, validation, observability yourself.
Tyk Open source Full-featured open-source alternative to Kong. Less plugin ecosystem.
The honest "why API Gateway"

On AWS, the value isn't that API Gateway is the best gateway — it's that it's already integrated with everything else you're using. Cognito for auth, Lambda for handlers, WAF for protection, CloudWatch for logs, IAM for access control, ACM for certs. Reaching for Kong or NGINX means wiring those integrations yourself. For most teams that's not worth it. If you're multi-cloud, multi-region active-active, or burning $50k+/month on API Gateway, look at Kong or Cloudflare.

05 Who actually runs it for you?

Like DynamoDB, API Gateway is fully managed — no servers to patch, no version upgrades. The interesting work is API design, auth patterns, and cost discipline.

👤 Self-managed (solo / startup)

You define routes via Terraform, SAM, or CDK. Most learners start here. The default for any solo serverless project.

🛠️ Platform team

Wraps API Gateway in a reusable Terraform module. Standard auth, logging, WAF, alarms baked in. App teams declare a route via PR.

🏗️ API design specialists

At larger orgs, API design is owned by an API platform team that enforces OpenAPI standards, versioning, and a developer portal.

🛡️ Security / AppSec

Owns the WAF rule sets, mTLS configurations, authorizer logic, and pen-test schedule. API Gateway is the front line of public-facing security.

💰 FinOps

API Gateway gets expensive. Watches per-route invocation counts; flags candidates for caching, throttling, or migrating to ALB at scale.

🤖 SaaS gateway services

Companies sometimes layer Cloudflare, Akamai, or Apigee in front of API Gateway for global edge + better analytics. Hybrid pattern.

⚠️ Cost gotcha

API Gateway is cheap until it isn't. HTTP API at $1.00 per million calls sounds free, but a moderately busy public API (1,000 RPS sustained) is 86 million calls/day = ~$2,580/month just on the gateway, not counting Lambda or data transfer. At that scale, ALB + ECS Fargate is often dramatically cheaper. Profile your actual traffic before assuming serverless is "cheaper."

06 Security mindset

APIs are the #1 attack surface for modern apps. The OWASP API Security Top 10 (2023) is required reading — it catalogs the API-specific failure modes (broken object-level auth, broken auth, excessive data exposure, etc.) that traditional web vuln lists miss. Map every API Gateway control below to a Top-10 entry.

Five things to internalize before shipping a public API

  • Always require an authorizer. Even for "internal" APIs. Pick the right one: Cognito for user-auth, JWT for OIDC providers, Lambda authorizer for custom logic, IAM for service-to-service. Never leave AuthorizationType=NONE in production by accident — it's how data gets scraped.
  • Set per-stage and per-method throttling. Default account limit is 10,000 RPS — but a runaway client can still chew through your Lambda budget in minutes. Throttle per-route, and use API keys + usage plans for partner-tier limits.
  • Validate request bodies with JSON Schema. Rejects malformed payloads at the gateway before they hit your Lambda — saves money and kills a class of injection attacks at the door.
  • Bolt on AWS WAF for public APIs. Managed rule sets (AWSManagedRulesCommonRuleSet, AWSManagedRulesKnownBadInputsRuleSet, AWSManagedRulesAmazonIpReputationList) catch the obvious junk for ~$5/month + per-request fees.
  • Enable execution logs and X-Ray. Default logging is light; execution logs (full request/response) are off by default. Turn them on for sensitive APIs — and protect the log group with KMS, because logs will contain auth tokens you shouldn't have.
For your career

API Gateway shows up on the AWS Solutions Architect, Developer, and Security Specialty exams. The recurring questions: which authorizer for which use case (Cognito vs Lambda vs IAM), public vs private API endpoints, how to rate-limit per consumer (API keys + usage plans), and integration with WAF. Combined with the OWASP API Top 10, this is high-leverage cloud-security knowledge.
