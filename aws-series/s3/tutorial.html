S3: Ship a Hardened Bucket

Three tracks, same lab. We build a private-by-default bucket the way you'd actually ship it in production — encrypted, versioned, logged, and auditable. No public toggles. No legacy ACLs. No surprise bills.

What's in this tutorial

  1. Prerequisites & safety net
  2. What we'll build

  4. Track B — AWS CLI v2
  5. Track C — Terraform

01 Prerequisites & safety net

You'll need

  • An AWS account (S3 free tier: 5 GB storage, 20K GET, 2K PUT/month for 12 months — this lab uses pennies of it).
  • An IAM user or SSO identity. Don't use the root account.
  • AWS CLI v2 installed and configured with a named profile (aws --version should print 2.x).
  • Terraform 1.7+ for Track C.

Pick a globally unique bucket name

Bucket names are globally unique across every AWS account in the world. my-bucket was claimed in 2006. Use a personal prefix: 

# Replace 'eddie' with something unique to you, and keep it lowercase.
# Bucket names: 3–63 chars, lowercase, digits, hyphens. No underscores, no dots
# (dots break TLS wildcard certs and CloudFront).
export BUCKET=eddie-aws-series-$(date +%Y%m%d)
echo "$BUCKET"

Set the AWS profile + region for every command

Every CLI command in this tutorial uses --profile eddie-lab --region us-east-1. Adjust to your profile/region.

⚠️ Check Block Public Access at the account level — once

Before doing anything else: in the S3 Console go to Block Public Access settings for this account and make sure all four account-level switches are ON. This is your global kill-switch — even if you misconfigure a bucket later, this prevents it from going public. Set it once per account; leave it on.

02 What we'll build

One private S3 bucket configured the way a security-minded team would actually ship it. Same outcome across all three tracks:

  • Bucket-level Block Public Access — all four switches ON.
  • Object Ownership = Bucket Owner Enforced — ACLs disabled. Bucket policies + IAM only.
  • Default encryption — SSE-S3 (AES-256). For "real" workloads we'd swap to SSE-KMS with a customer-managed key; we note that path but don't create the key in this lab to keep cleanup simple.
  • Versioning enabled — every overwrite/delete is recoverable.
  • Lifecycle rule — non-current versions expire after 30 days; aborted multipart uploads cleaned up after 7 days (saves silent storage cost).
  • Server access logging enabled to a separate logging bucket.
  • Bucket policy that denies any non-TLS request and denies any unencrypted PUT — defense in depth on top of the default-encryption setting.
Why these defaults

Every line above is a control that maps directly to the CIS AWS Foundations Benchmark and to AWS Security Hub findings. If you ship buckets like this from day one, your first compliance audit is largely a paperwork exercise instead of a remediation marathon.

Track A Console — click-by-click

Best the first time you ship a hardened bucket. Every tickbox you'll touch matters.

  1. Sign in at console.aws.amazon.com as your IAM user (not root). Pick region US East (N. Virginia) us-east-1 top right.
  2. Create the logging bucket first. Go to S3 → Create bucket.
    • Bucket name: eddie-aws-series-YYYYMMDD-logs (your prefix, today's date, -logs suffix).
    • Object Ownership: ACLs disabled (Bucket owner enforced).
    • Block Public Access: Block all public access (default; leave on).
    • Bucket Versioning: Disable (logs don't need it; saves cost).
    • Default encryption: SSE-S3 (Amazon S3 managed keys).
    • Tags: Project = aws-series-tutorial.
    • Create bucket.
  3. Create the data bucket. S3 → Create bucket again.
    • Bucket name: eddie-aws-series-YYYYMMDD.
    • Object Ownership: ACLs disabled (Bucket owner enforced).
    • Block Public Access: leave all four switches ON.
    • Bucket Versioning: Enable.
    • Default encryption: SSE-S3 (Amazon S3 managed keys). (For SSE-KMS, swap and pick a KMS key — see takeaways.)
    • Advanced → Object Lock: leave disabled for this lab. (Enable it for compliance / WORM workloads — but it's permanent.)
    • Tags: Project = aws-series-tutorial.
    • Create bucket.
  4. Wire up server access logging on the data bucket. Open the data bucket → Properties → Server access logging → Edit.
    • Enable.
    • Target bucket: s3://eddie-aws-series-YYYYMMDD-logs/access-logs/.
    • Save changes.
  5. Add the lifecycle rule. Data bucket → Management → Lifecycle rules → Create lifecycle rule.
    • Name: expire-old-versions-and-multipart.
    • Scope: Apply to all objects in the bucket.
    • Permanently delete noncurrent versions after 30 days.
    • Delete expired delete markers.
    • Delete incomplete multipart uploads after 7 days.
    • Create rule.
  6. Attach a TLS-only / encryption-required bucket policy. Data bucket → Permissions → Bucket policy → Edit. Paste: 
    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyInsecureTransport",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::eddie-aws-series-YYYYMMDD",
        "arn:aws:s3:::eddie-aws-series-YYYYMMDD/*"
      ],
      "Condition": { "Bool": { "aws:SecureTransport": "false" } }
    },
    {
      "Sid": "DenyUnencryptedPuts",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::eddie-aws-series-YYYYMMDD/*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": [ "AES256", "aws:kms" ]
        }
      }
    }
  ]
}

    Replace both ARNs with your actual bucket name. Save changes.

  7. Verify. Skip ahead to "Verify & exercise" for the smoke tests.

Track B AWS CLI v2

Same lab from your terminal. Worth doing once — every Console toggle maps to an API call here.

1. Set names and create both buckets

export PROFILE=eddie-lab
export REGION=us-east-1
export BUCKET=eddie-aws-series-$(date +%Y%m%d)
export LOG_BUCKET="${BUCKET}-logs"

# us-east-1 is special: don't pass --create-bucket-configuration.
# In any other region you'd add: --create-bucket-configuration LocationConstraint=$REGION
aws s3api create-bucket --bucket "$LOG_BUCKET" \
  --region "$REGION" --profile "$PROFILE"

aws s3api create-bucket --bucket "$BUCKET" \
  --region "$REGION" --profile "$PROFILE"

2. Lock down ownership + public access on both

for B in "$BUCKET" "$LOG_BUCKET"; do
  aws s3api put-bucket-ownership-controls \
    --bucket "$B" \
    --ownership-controls 'Rules=[{ObjectOwnership=BucketOwnerEnforced}]' \
    --profile "$PROFILE"

  aws s3api put-public-access-block \
    --bucket "$B" \
    --public-access-block-configuration \
        BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true \
    --profile "$PROFILE"
done

3. Default encryption (SSE-S3) on both

for B in "$BUCKET" "$LOG_BUCKET"; do
  aws s3api put-bucket-encryption \
    --bucket "$B" \
    --server-side-encryption-configuration '{
      "Rules": [{
        "ApplyServerSideEncryptionByDefault": { "SSEAlgorithm": "AES256" },
        "BucketKeyEnabled": true
      }]
    }' \
    --profile "$PROFILE"
done

4. Versioning on the data bucket only

aws s3api put-bucket-versioning \
  --bucket "$BUCKET" \
  --versioning-configuration Status=Enabled \
  --profile "$PROFILE"

5. Lifecycle rule (expire old versions + abort hung multipart)

cat > /tmp/lifecycle.json <<'EOF'
{
  "Rules": [
    {
      "ID": "expire-old-versions-and-multipart",
      "Status": "Enabled",
      "Filter": {},
      "NoncurrentVersionExpiration": { "NoncurrentDays": 30 },
      "AbortIncompleteMultipartUpload": { "DaysAfterInitiation": 7 },
      "Expiration": { "ExpiredObjectDeleteMarker": true }
    }
  ]
}
EOF

aws s3api put-bucket-lifecycle-configuration \
  --bucket "$BUCKET" \
  --lifecycle-configuration file:///tmp/lifecycle.json \
  --profile "$PROFILE"

6. Server access logging → logging bucket

The logging bucket needs a policy that lets the S3 logging service principal write to it.

cat > /tmp/log-bucket-policy.json <<EOF
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "AllowS3LoggingWrite",
    "Effect": "Allow",
    "Principal": { "Service": "logging.s3.amazonaws.com" },
    "Action": "s3:PutObject",
    "Resource": "arn:aws:s3:::${LOG_BUCKET}/access-logs/*",
    "Condition": {
      "StringEquals": { "aws:SourceAccount": "$(aws sts get-caller-identity --query Account --output text --profile $PROFILE)" }
    }
  }]
}
EOF

aws s3api put-bucket-policy \
  --bucket "$LOG_BUCKET" \
  --policy file:///tmp/log-bucket-policy.json \
  --profile "$PROFILE"

aws s3api put-bucket-logging \
  --bucket "$BUCKET" \
  --bucket-logging-status "{
    \"LoggingEnabled\": {
      \"TargetBucket\": \"$LOG_BUCKET\",
      \"TargetPrefix\": \"access-logs/\"
    }
  }" \
  --profile "$PROFILE"

7. TLS-only + encryption-required policy on the data bucket

cat > /tmp/data-bucket-policy.json <<EOF
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyInsecureTransport",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::${BUCKET}",
        "arn:aws:s3:::${BUCKET}/*"
      ],
      "Condition": { "Bool": { "aws:SecureTransport": "false" } }
    },
    {
      "Sid": "DenyUnencryptedPuts",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::${BUCKET}/*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": ["AES256", "aws:kms"]
        }
      }
    }
  ]
}
EOF

aws s3api put-bucket-policy \
  --bucket "$BUCKET" \
  --policy file:///tmp/data-bucket-policy.json \
  --profile "$PROFILE"

Now jump to Verify & exercise the bucket.

Track C Terraform

The production track. Idempotent, code-reviewable, destroy-safe.

1. Working directory

mkdir -p ~/labs/s3-tutorial && cd ~/labs/s3-tutorial

Save this as main.tf. Replace the name_prefix with something globally unique to you.

terraform {
  required_version = ">= 1.7"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region  = "us-east-1"
  profile = "eddie-lab"

  default_tags {
    tags = {
      Project   = "aws-series-tutorial"
      ManagedBy = "terraform"
    }
  }
}

variable "name_prefix" {
  description = "Globally unique prefix — must not collide with any other AWS account"
  type        = string
  default     = "eddie-aws-series"
}

locals {
  data_bucket = "${var.name_prefix}-${formatdate("YYYYMMDD", timestamp())}"
  log_bucket  = "${local.data_bucket}-logs"
}

data "aws_caller_identity" "me" {}

# --------------------------------------------------------------------
# Logging bucket
# --------------------------------------------------------------------
resource "aws_s3_bucket" "logs" {
  bucket        = local.log_bucket
  force_destroy = true # easier teardown for a lab; remove for prod
}

resource "aws_s3_bucket_ownership_controls" "logs" {
  bucket = aws_s3_bucket.logs.id
  rule { object_ownership = "BucketOwnerEnforced" }
}

resource "aws_s3_bucket_public_access_block" "logs" {
  bucket                  = aws_s3_bucket.logs.id
  block_public_acls       = true
  ignore_public_acls      = true
  block_public_policy     = true
  restrict_public_buckets = true
}

resource "aws_s3_bucket_server_side_encryption_configuration" "logs" {
  bucket = aws_s3_bucket.logs.id
  rule {
    apply_server_side_encryption_by_default { sse_algorithm = "AES256" }
    bucket_key_enabled = true
  }
}

# Allow the S3 logging service to deliver access logs into this bucket
resource "aws_s3_bucket_policy" "logs" {
  bucket = aws_s3_bucket.logs.id
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Sid       = "AllowS3LoggingWrite"
      Effect    = "Allow"
      Principal = { Service = "logging.s3.amazonaws.com" }
      Action    = "s3:PutObject"
      Resource  = "${aws_s3_bucket.logs.arn}/access-logs/*"
      Condition = {
        StringEquals = { "aws:SourceAccount" = data.aws_caller_identity.me.account_id }
      }
    }]
  })
}

# --------------------------------------------------------------------
# Data bucket
# --------------------------------------------------------------------
resource "aws_s3_bucket" "data" {
  bucket        = local.data_bucket
  force_destroy = true # easier teardown for a lab; remove for prod
}

resource "aws_s3_bucket_ownership_controls" "data" {
  bucket = aws_s3_bucket.data.id
  rule { object_ownership = "BucketOwnerEnforced" }
}

resource "aws_s3_bucket_public_access_block" "data" {
  bucket                  = aws_s3_bucket.data.id
  block_public_acls       = true
  ignore_public_acls      = true
  block_public_policy     = true
  restrict_public_buckets = true
}

resource "aws_s3_bucket_server_side_encryption_configuration" "data" {
  bucket = aws_s3_bucket.data.id
  rule {
    apply_server_side_encryption_by_default { sse_algorithm = "AES256" }
    bucket_key_enabled = true
  }
}

resource "aws_s3_bucket_versioning" "data" {
  bucket = aws_s3_bucket.data.id
  versioning_configuration { status = "Enabled" }
}

resource "aws_s3_bucket_lifecycle_configuration" "data" {
  bucket = aws_s3_bucket.data.id

  rule {
    id     = "expire-old-versions-and-multipart"
    status = "Enabled"
    filter {}

    noncurrent_version_expiration { noncurrent_days = 30 }
    abort_incomplete_multipart_upload { days_after_initiation = 7 }
    expiration { expired_object_delete_marker = true }
  }

  depends_on = [aws_s3_bucket_versioning.data]
}

resource "aws_s3_bucket_logging" "data" {
  bucket        = aws_s3_bucket.data.id
  target_bucket = aws_s3_bucket.logs.id
  target_prefix = "access-logs/"

  depends_on = [aws_s3_bucket_policy.logs]
}

# Defense in depth on top of default encryption: deny non-TLS and unencrypted PUTs
resource "aws_s3_bucket_policy" "data" {
  bucket = aws_s3_bucket.data.id
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid       = "DenyInsecureTransport"
        Effect    = "Deny"
        Principal = "*"
        Action    = "s3:*"
        Resource  = [aws_s3_bucket.data.arn, "${aws_s3_bucket.data.arn}/*"]
        Condition = { Bool = { "aws:SecureTransport" = "false" } }
      },
      {
        Sid       = "DenyUnencryptedPuts"
        Effect    = "Deny"
        Principal = "*"
        Action    = "s3:PutObject"
        Resource  = "${aws_s3_bucket.data.arn}/*"
        Condition = {
          StringNotEquals = {
            "s3:x-amz-server-side-encryption" = ["AES256", "aws:kms"]
          }
        }
      }
    ]
  })

  depends_on = [aws_s3_bucket_public_access_block.data]
}

output "data_bucket"    { value = aws_s3_bucket.data.bucket }
output "logging_bucket" { value = aws_s3_bucket.logs.bucket }

2. Init, plan, apply

terraform init
terraform plan      # READ THIS. Always read the plan before applying.
terraform apply     # type 'yes' to confirm

The output prints both bucket names. Continue to Verify & exercise.

Why Terraform splits S3 into many resources

Modern AWS provider versions split aws_s3_bucket into a dozen sub-resources (aws_s3_bucket_versioning, _lifecycle_configuration, _public_access_block, etc.). It looks verbose, but it prevents drift: each setting becomes its own diff in terraform plan, which is what you actually want during code review.

06 Verify & exercise the bucket

One set of commands proves the controls work. Replace $BUCKET with your data bucket name (or export it first).

Upload an object the right way (TLS, default encryption)

echo "hello s3" > /tmp/hello.txt

aws s3 cp /tmp/hello.txt "s3://$BUCKET/hello.txt" \
  --profile eddie-lab

# Read it back
aws s3 cp "s3://$BUCKET/hello.txt" - --profile eddie-lab

Prove versioning works

echo "hello s3 v2" > /tmp/hello.txt
aws s3 cp /tmp/hello.txt "s3://$BUCKET/hello.txt" --profile eddie-lab

aws s3api list-object-versions \
  --bucket "$BUCKET" \
  --prefix hello.txt \
  --query 'Versions[].{VersionId:VersionId,IsLatest:IsLatest,LastModified:LastModified}' \
  --output table \
  --profile eddie-lab

Prove the bucket is not public

# This should return a 403 / AccessDenied — no anonymous reads allowed.
curl -s -o /dev/null -w "%{http_code}\n" "https://$BUCKET.s3.amazonaws.com/hello.txt"

Prove TLS-only enforcement

# Plain HTTP request → bucket policy denies it (you'll get 400/403).
curl -s -o /dev/null -w "%{http_code}\n" "http://$BUCKET.s3.amazonaws.com/hello.txt"

Prove unencrypted-PUT denial

Default encryption already encrypts every PUT, so to actually trigger the deny you'd need to send x-amz-server-side-encryption: anything-else. Quick simulation via AWS CLI debug logs is enough — for most learners, reading the bucket policy carefully is the lesson.

Watch CloudTrail / access logs catch up

Server access logs are best-effort and can take minutes to appear. After a few minutes:

aws s3 ls "s3://${BUCKET}-logs/access-logs/" --profile eddie-lab

Each log file is one S3-formatted line per request. CloudTrail data events (not enabled by default) give you near-real-time, IAM-attributed records — turn those on in CloudTrail when you want full request-level audit.

07 Cleanup — read this first

Why this matters

S3 storage cost is small ($0.023/GB-month standard), but a forgotten bucket with versioning + lifecycle disabled can grow forever. Versioned buckets need special handling — you have to delete every object version, not just the latest, before the bucket itself can be removed. Terraform's force_destroy = true handles this for you; the CLI/Console paths don't.

Track C (Terraform) — easiest

cd ~/labs/s3-tutorial
terraform destroy   # type 'yes' — wipes everything we created (including versions)

Track B (CLI) — manual, versioning-aware

# Empty all versions + delete markers from the data bucket
aws s3api list-object-versions \
  --bucket "$BUCKET" \
  --output json \
  --profile eddie-lab \
| jq '{Objects: ([.Versions[]?, .DeleteMarkers[]?] | map({Key, VersionId})), Quiet: false}' \
> /tmp/del.json

# Only call delete-objects if there's anything to delete
if [ "$(jq '.Objects | length' /tmp/del.json)" -gt 0 ]; then
  aws s3api delete-objects \
    --bucket "$BUCKET" \
    --delete file:///tmp/del.json \
    --profile eddie-lab
fi

aws s3api delete-bucket --bucket "$BUCKET"     --profile eddie-lab

# Logging bucket — versioning is off, so plain rm works
aws s3 rm "s3://$LOG_BUCKET" --recursive --profile eddie-lab
aws s3api delete-bucket --bucket "$LOG_BUCKET" --profile eddie-lab

Track A (Console)

  1. Open the data bucket → Empty button → type permanently delete → confirm.
  2. Then Delete the (now empty) bucket.
  3. Repeat for the -logs bucket.

Verify nothing is left

aws s3 ls --profile eddie-lab | grep eddie-aws-series
# (empty output = clean)

08 Senior-engineer takeaways

What this exercise drilled into you

  • Private by default, always. BPA + Bucket Owner Enforced + no public policy = the right starting point. Public S3 should be a deliberate, reviewed exception, never a default.
  • Defense in depth. Default encryption + the DenyUnencryptedPuts condition + KMS for sensitive workloads = three independent layers. Removing one shouldn't expose data.
  • Versioning is your ransomware insurance — but it costs. Without a lifecycle rule, deleted/overwritten versions accumulate forever. The 30-day expire rule is the standard tradeoff.
  • Logs go to a separate bucket. Same-bucket logging creates infinite recursion ("the log of the log of the log"). Different bucket, different lifecycle.
  • TLS-only is one bucket-policy condition. aws:SecureTransport=false in a Deny statement is a one-line control that auditors love and almost everyone forgets.

Where to take it next

  • Swap SSE-S3 for SSE-KMS with a customer-managed key. Now you control who can decrypt — even root can't read without the key. Required for HIPAA/PCI in practice.
  • Front the bucket with CloudFront + Origin Access Control (OAC). The bucket stays private; CloudFront fetches with a signed identity. This is the modern static-site / asset-CDN pattern.
  • Enable CloudTrail data events for the bucket so every GetObject / PutObject is logged with the calling IAM identity.
  • Turn on IAM Access Analyzer for S3 at the account level. Free, continuous, and the highest-leverage alert for the time spent.
  • For compliance / WORM workloads, enable Object Lock at bucket creation (it's permanent — pick a default retention period carefully).
  • For multi-region resilience, add Cross-Region Replication (CRR) to a bucket in a second region. Pairs well with KMS multi-region keys.

Costs at a glance (us-east-1, May 2026 list price)

  • S3 Standard storage: $0.023/GB-month. 5 GB free for 12 months.
  • PUT/COPY/POST/LIST: $0.005 per 1,000 requests. 2K free/month.
  • GET/SELECT: $0.0004 per 1,000 requests. 20K free/month.
  • Egress to internet: $0.09/GB (this is the bill killer at scale — that's why R2/B2 exist).
  • Server access logging: free (you pay for the storage of the log objects, which is tiny).
  • SSE-S3: free. SSE-KMS: $0.03 per 10,000 requests + KMS key fee ($1/month).

Pricing reference: aws.amazon.com/s3/pricing. Regions vary; verify before quoting.

Done?

You've shipped a private, encrypted, versioned, logged, TLS-enforced bucket across three tracks. That's a real production-grade S3 baseline — the foundation every later service in this series will sit on top of.
