Amazon S3

Simple Storage Service — durable object storage that the entire internet runs on. Eleven nines of durability, infinite scale, pennies per gigabyte — and the most-misconfigured service in cloud security history.

01 What is S3?

S3 stores files — billions of them, of any size up to 5 TB each — as objects inside buckets, accessed over HTTPS via a globally unique URL. AWS designs it for 11 nines of durability (99.999999999%) and practically unlimited scale.

Launched in March 2006 as the very first public AWS service, S3 predates EC2 by months. Today it stores hundreds of trillions of objects, peaks at tens of millions of requests per second, and underpins a huge fraction of the internet — websites, backups, data lakes, ML training sets, video archives, container images, application logs. If you've used the web today, you've read from S3.

The core mental model

  • Object storage, not a filesystem. No directories. Keys like logs/2026/05/01/app.log look like paths but are flat strings. Operations are PUT, GET, DELETE, LIST — not open() / seek().
  • Bucket name = global namespace. Bucket names are unique across every AWS account on Earth. my-bucket was taken in 2006. Use a prefix like eddie-2026-logs.
  • Eventually consistent → strongly consistent (since Dec 2020). A successful PUT is immediately readable. This used to not be true; old blog posts may still warn about it. Ignore them.
  • Storage classes are a first-class concept. Standard, Infrequent Access, One-Zone-IA, Glacier Instant / Flexible / Deep Archive, Intelligent-Tiering. Same API — different price/latency tradeoff.
  • Versioning + Object Lock turn S3 into a tamper-evident, ransomware-resistant store — required by many compliance regimes (SEC 17a-4, HIPAA backup rules).
  • Eventual cost surprises hide in LIST calls and cross-region transfer. Storage is cheap. Egress and request volume aren't.
In one sentence

S3 is the internet's hard drive — durable, infinitely scalable, addressable via HTTPS, and so cheap to keep data in that the actual cost question is almost always "how often will you read it?" not "how much can you store?"

02 A vibrant example: Dropbox, then and now

In 2008, Dropbox launched on top of S3. They didn't have to build storage — they uploaded your file to a bucket, kept the metadata in their own database, and let S3 do the heavy lifting. That's how a small team scaled to hundreds of millions of users without owning a single hard drive.

Years later, at sufficient scale, Dropbox famously moved off S3 onto their own "Magic Pocket" infrastructure — they were big enough that the unit economics finally favored building. That is the S3 success story: the only reason to ever leave S3 is that you've outgrown the entire public cloud pricing model. Almost no one does.

Today the same pattern repeats every week: a startup ships a "we host your photos / videos / contracts / X-rays" product backed by a single S3 bucket, a Lambda for thumbnails, and CloudFront in front for delivery. Two engineers, infinite storage, $30/month bill at launch.

03 Who uses S3 — and why

Effectively everyone with an AWS account, plus a huge number of customers using S3 standalone for storage. Concrete examples (publicly documented):

📸 Pinterest

Hundreds of billions of pins — every image you've ever pinned lives in S3, served via CloudFront. Pinterest Engineering →

🎬 Netflix

Source video masters, encoded variants, subtitles, manifests — all sit in S3 before being pushed to Open Connect edges.

🛰️ NASA / NOAA

Petabytes of open satellite imagery (Landsat, GOES, Sentinel) on S3 as part of the AWS Open Data program. Free to read in-region.

🧬 Genomics England, Illumina

Whole-genome sequences are gigabytes each. S3 + Glacier holds the cohort archive; EC2 / Batch reads it for analysis runs.

🏛️ U.S. Government

FedRAMP-authorized S3 in GovCloud holds sensitive but unclassified data for DoD, DHS, and intelligence agency workloads.

🤖 OpenAI / Anthropic / model trainers

Multi-petabyte training corpora, model checkpoints, eval datasets — S3 is the standard ML data plane (often paired with FSx for Lustre for hot reads).

Common reasons to reach for S3

  • Static website / SPA hosting: bucket + CloudFront + Route 53 = global, HTTPS, sub-cent per GB. The default frontend deploy target for half the internet.
  • Backups & archives: nightly snapshots, log retention, S3 + Glacier Deep Archive at $1/TB/month. Cheaper than tape — and online.
  • Data lake: Parquet / ORC files in S3 queried by Athena, Redshift Spectrum, EMR, Spark. The standard analytics storage layer.
  • Application uploads: presigned URLs let your app accept user uploads directly to S3 without proxying through your servers. Saves bandwidth, scales infinitely.
  • Inter-service handoff: Service A writes a file, S3 fires an EventBridge event, Lambda B picks it up. The original serverless ETL pattern.
  • Compliance retention: Object Lock + WORM modes give you legal-hold and retention-period guarantees that auditors recognize.

04 Competitors & market share

Object storage is a commodity primitive — every cloud has one, and a few independents specifically target S3's egress pricing. Honest framing: AWS defined the API, and almost everyone else implements an "S3-compatible" surface.

Provider Service S3 API compatible? Where it wins / loses
AWS S3 The reference Most storage classes, deepest IAM, biggest ecosystem (Athena, Glue, Lake Formation, Snowball). Loss: egress pricing is famously expensive ($0.09/GB out to internet in most regions).
Microsoft Azure Blob Storage No (different API) Hot / Cool / Archive tiers, AAD integration, integrated with Synapse and Fabric. Strong for Microsoft-heavy enterprises.
Google Cloud Cloud Storage (GCS) Partial (interop layer) Excellent multi-region buckets, single global namespace per bucket. Often pricing-competitive, especially analytics with BigQuery.
Cloudflare R2 Yes Zero egress fees. Game-changer for bandwidth-heavy workloads (video, downloads). Smaller storage-class catalog and weaker analytics integration.
Backblaze B2 Yes ~$6/TB/month, very cheap egress to Cloudflare. Popular for backup workflows. Smaller ecosystem.
Wasabi Hot Cloud Storage Yes Flat pricing, no egress fees within fair-use limits. Pitched as "S3 with predictable bills."
MinIO, Ceph, OpenStack Swift Self-hosted Yes (MinIO/Ceph) Run S3-compatible storage on your own hardware — common for on-prem AI clusters and air-gapped environments.

Pricing claims above are headline figures; verify on each provider's pricing page before quoting in a proposal.

The honest "why AWS S3"

S3 wins because the ecosystem wins. Every analytics tool, every backup product, every CI/CD runner has S3 as the default storage backend. The switching cost isn't moving the bytes — it's rewriting the integrations. For pure bandwidth-heavy workloads (video, large downloads, public datasets), Cloudflare R2 with zero egress can be a dramatic cost win and is worth a serious look.

Where AWS doesn't always win

  • Egress-heavy products: if you serve many TB/month to end users, R2 / B2 will absolutely beat S3 on bill.
  • Single global bucket: GCS gives you a multi-region bucket out of the box; S3 requires cross-region replication (extra storage cost).
  • Predictable flat pricing: Wasabi and similar appeal to teams that hate surprise request and egress charges.
  • On-prem / air-gapped: MinIO and Ceph let you run an S3 API where AWS literally cannot reach.

05 Who actually runs it for you?

S3's shared-responsibility line is unusually skewed in AWS's favor: AWS runs the storage, replication, durability math, and the API front door. You own the bucket policy, the encryption choice, the public-access settings, and the lifecycle rules. Almost every "S3 breach" you've read about was the customer side, not the AWS side.

👤 Self-managed (solo / startup)

You create buckets, write policies, and (hopefully) enable Block Public Access. Easy to ship; easy to misconfigure. The default lab setup.

🛠️ Platform team

At scale a platform team ships a Terraform module ("the blessed bucket"): private by default, KMS-encrypted, versioned, logged, lifecycled. App teams just call the module.

🏢 AWS Organizations + SCPs

Service Control Policies block account-wide footguns: deny s3:PutBucketPublicAccessBlock with disable, deny s3:PutBucketAcl grants to AllUsers. Guardrails, not requests.

📡 Macie / Access Analyzer

AWS Macie continuously scans S3 for PII; IAM Access Analyzer flags buckets with cross-account or public access. Often run by the security team, not the app team.

🤝 Backup / archive vendors

Veeam, Rubrik, Druva, AWS Backup all use S3 (and Glacier) under the hood for enterprise backup. You consume their UI; the bucket is theirs.

🤖 Higher-level abstractions

Amplify Storage, AppSync file uploads, Lambda + presigned URLs — frameworks abstract away the bucket lifecycle. You may never touch the S3 console directly.

⚠️ The default isn't neutral anymore

Since April 2023, new S3 buckets have Block Public Access on by default and ACLs disabled. That's a huge improvement — but older buckets in your account may not have those defaults. Always audit existing buckets; don't assume the modern defaults apply retroactively.

06 Security mindset

Misconfigured S3 is the headline cloud breach. A short, incomplete list of famous public-bucket exposures: Verizon (2017, 14M customer records), Accenture (2017, internal infrastructure), Pentagon / U.S. Army INSCOM (2017, classified-adjacent data), Capital One (2019 — more famously an EC2/IAM flaw, but the data lived in S3), GoDaddy (2018, server config), Booz Allen Hamilton (2017, geospatial intelligence). The pattern almost never changes: a bucket got marked public and nobody noticed until a researcher tweeted.

The five S3 controls every security engineer must know cold

Control What it does When it fires
Block Public Access (BPA) Account- and bucket-level switches that override any policy or ACL granting public read/write. The kill switch. Turn it on at the account level. Unless you are explicitly hosting public files, leave it on.
Bucket Policy Resource-based JSON policy attached to the bucket. Controls who can do what — most powerful S3 access control. Default access decision when IAM policies and bucket policies disagree → most-restrictive wins.
Default Encryption (SSE-S3 / SSE-KMS) Every new object encrypted at rest. SSE-KMS gives per-object audit logs and customer-managed keys. Always on by default since Jan 2023. Customer-managed KMS keys cost extra but give full key control.
Versioning + Object Lock Versioning keeps every overwrite/delete recoverable. Object Lock prevents deletion entirely (WORM). Ransomware defense. A compromised key can't permanently delete what Object Lock retains.
Access Logging + CloudTrail Data Events Server access logs (every request, ~1-min latency) + CloudTrail data events (every GetObject/PutObject, near real time, queryable by IAM identity). Required for incident response and most compliance frameworks. CloudTrail data events are not on by default and cost extra.

Six things to internalize before your first production bucket

  • Account-level Block Public Access — set this once, leave it on. It's a global override that prevents anyone in the account from publishing a bucket by accident.
  • Disable ACLs (Object Ownership = Bucket Owner Enforced). ACLs are a 2006-era mechanism and the source of most legacy public-bucket bugs. Use bucket policies + IAM exclusively.
  • Encrypt with KMS for sensitive data. SSE-S3 protects against physical-disk theft. SSE-KMS adds per-object access decisions and full audit logs — required for HIPAA / PCI / SOC 2 in practice.
  • Enable versioning everywhere it doesn't bankrupt you. Pair with a lifecycle rule that expires non-current versions after 30/90 days. Versioning + lifecycle = ransomware insurance.
  • Use Origin Access Control (OAC) with CloudFront, never public read. If a file should be served to the public web, the bucket itself stays private and CloudFront fetches it with a signed identity. This is now the AWS-recommended pattern.
  • Enable IAM Access Analyzer for S3. Free, account-wide, continuously flags any bucket with public or cross-account access. The single highest-value alert pipeline you'll wire up.
For your career

S3 misconfiguration is on every cloud-security interview, every AWS Security Specialty exam, and every real incident-response playbook. Get fluent with: bucket policy syntax (especially Condition blocks), the BPA precedence model, KMS-vs-S3 encryption decisions, OAC, presigned URLs, and the difference between server access logs and CloudTrail data events. These come up constantly.
