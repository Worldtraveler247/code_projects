01 Prerequisites & safety net

You'll need

  • An AWS account (free-tier eligible — t3.micro is free for 750 hrs/month for 12 months).
  • An IAM user with programmatic access or AWS SSO/Identity Center configured. Don't use the root account.
  • AWS CLI v2 installed (aws --version should print 2.x).
  • Terraform 1.7+ for Track C.
  • Session Manager plugin for the AWS CLI (we'll use this to connect — no SSH keys).

Configure a named CLI profile (one-time setup)

Don't dump credentials into env vars; use named profiles in ~/.aws/credentials. Replace eddie-lab with whatever you want to call this profile.

aws configure --profile eddie-lab
# AWS Access Key ID [None]: AKIA...
# AWS Secret Access Key [None]: ...
# Default region name [None]: us-east-1
# Default output format [None]: json

# Verify
aws sts get-caller-identity --profile eddie-lab

Every CLI command in this tutorial uses --profile eddie-lab --region us-east-1. Adjust to your profile/region.

⚠️ Set a billing alarm BEFORE you launch anything

First-time learners regularly forget a t3.micro running, or accidentally launch a t3.large. A $50 alarm sends you an email before the bill spirals. Run this before you launch the instance:

First, create an SNS topic and subscribe your email:

# Create the SNS topic that the alarm will publish to
aws sns create-topic \
  --name billing-alerts \
  --region us-east-1 \
  --profile eddie-lab

# Subscribe your email (you'll get a confirmation email — click the link)
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:<ACCOUNT_ID>:billing-alerts \
  --protocol email \
  --notification-endpoint you@example.com \
  --region us-east-1 \
  --profile eddie-lab

Then create the alarm. Note: billing metrics only live in us-east-1, even if your workloads are elsewhere — that's an AWS quirk, not a typo.

aws cloudwatch put-metric-alarm \
  --alarm-name aws-monthly-billing-50 \
  --alarm-description "Alert when monthly estimated charges exceed \$50" \
  --metric-name EstimatedCharges \
  --namespace AWS/Billing \
  --statistic Maximum \
  --period 21600 \
  --evaluation-periods 1 \
  --threshold 50 \
  --comparison-operator GreaterThanThreshold \
  --dimensions Name=Currency,Value=USD \
  --alarm-actions arn:aws:sns:us-east-1:<ACCOUNT_ID>:billing-alerts \
  --region us-east-1 \
  --profile eddie-lab

You also need to enable "Receive Billing Alerts" in Billing Preferences — one-time toggle.

02 What we'll build

One t3.micro EC2 instance running Amazon Linux 2023, in the default VPC, with no SSH key and no inbound security group rules. We connect via AWS Systems Manager Session Manager — IAM-authenticated, fully audited, no port 22 open to the internet.

Why no SSH

SSH on 0.0.0.0/0 is the #1 source of EC2 compromise. Session Manager gives you a shell over IAM-signed AWS API calls — no inbound port at all. Every session is logged in CloudTrail. This is the modern, audit-friendly way and what you'll see on the AWS Security Specialty exam.

Track A Console — click-by-click

Best for the first time, when you want to see the abstractions visually.

  1. Sign in at console.aws.amazon.com as your IAM user (not root). Pick region US East (N. Virginia) us-east-1 top right.
  2. Create the IAM role for SSM. Go to IAM → Roles → Create role.
    • Trusted entity: AWS service → use case EC2.
    • Permissions: search for and attach AmazonSSMManagedInstanceCore.
    • Name: ec2-tutorial-ssm-role.
    • Create role.
  3. Launch the instance. Go to EC2 → Instances → Launch instances.
    • Name: aws-series-tutorial.
    • AMI: Amazon Linux 2023 (HVM), SSD Volume Type — free-tier eligible.
    • Instance type: t3.micro.
    • Key pair: Proceed without a key pair. Yes, we mean it.
    • Network settings → Edit → Auto-assign public IP: Disable (we don't need it for SSM).
    • Firewall (security groups): Create new, name aws-series-tutorial-sg, delete the default SSH inbound rule. Leave inbound rules empty. Outbound default (allow all) is fine.
    • Advanced details → IAM instance profile: ec2-tutorial-ssm-role.
    • Advanced details → Metadata version: V2 only (token required) — that's IMDSv2.
    • Tags: Project = aws-series-tutorial.
    • Launch.
  4. Wait until Status check: 2/2 checks passed (1–2 min). Note the Instance ID (i-0abc...).

Skip ahead to Connect to open a shell.

Track B AWS CLI v2

Same lab, all from your terminal. Worth doing once — you'll learn what the Console buttons actually call.

1. Find the latest Amazon Linux 2023 AMI ID

AMI IDs are region-specific and change with each AL2023 release. Don't hardcode — query.

AMI_ID=$(aws ec2 describe-images \
  --owners amazon \
  --filters \
    "Name=name,Values=al2023-ami-*-x86_64" \
    "Name=state,Values=available" \
  --query 'sort_by(Images, &CreationDate)[-1].ImageId' \
  --output text \
  --region us-east-1 \
  --profile eddie-lab)

echo "Latest AL2023 AMI: $AMI_ID"

2. Create the IAM role + instance profile for SSM

# Trust policy (lets EC2 assume the role)
cat > /tmp/ec2-trust.json <<'EOF'
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "Service": "ec2.amazonaws.com" },
    "Action": "sts:AssumeRole"
  }]
}
EOF

aws iam create-role \
  --role-name ec2-tutorial-ssm-role \
  --assume-role-policy-document file:///tmp/ec2-trust.json \
  --profile eddie-lab

aws iam attach-role-policy \
  --role-name ec2-tutorial-ssm-role \
  --policy-arn arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore \
  --profile eddie-lab

aws iam create-instance-profile \
  --instance-profile-name ec2-tutorial-ssm-profile \
  --profile eddie-lab

aws iam add-role-to-instance-profile \
  --instance-profile-name ec2-tutorial-ssm-profile \
  --role-name ec2-tutorial-ssm-role \
  --profile eddie-lab

3. Create the security group (no inbound rules)

VPC_ID=$(aws ec2 describe-vpcs \
  --filters "Name=is-default,Values=true" \
  --query 'Vpcs[0].VpcId' \
  --output text \
  --region us-east-1 \
  --profile eddie-lab)

SG_ID=$(aws ec2 create-security-group \
  --group-name aws-series-tutorial-sg \
  --description "EC2 tutorial — no inbound, SSM only" \
  --vpc-id "$VPC_ID" \
  --query 'GroupId' \
  --output text \
  --region us-east-1 \
  --profile eddie-lab)

echo "SG: $SG_ID"

4. Launch the instance

INSTANCE_ID=$(aws ec2 run-instances \
  --image-id "$AMI_ID" \
  --instance-type t3.micro \
  --security-group-ids "$SG_ID" \
  --iam-instance-profile Name=ec2-tutorial-ssm-profile \
  --metadata-options "HttpTokens=required,HttpEndpoint=enabled" \
  --tag-specifications \
    'ResourceType=instance,Tags=[{Key=Name,Value=aws-series-tutorial},{Key=Project,Value=aws-series-tutorial}]' \
  --query 'Instances[0].InstanceId' \
  --output text \
  --region us-east-1 \
  --profile eddie-lab)

echo "Instance: $INSTANCE_ID"

# Wait for it to be running and pass status checks
aws ec2 wait instance-status-ok \
  --instance-ids "$INSTANCE_ID" \
  --region us-east-1 \
  --profile eddie-lab

Now skip to Connect.

Track C Terraform

The production track. Idempotent, code-reviewable, destroy-safe.

1. Create a working directory and main.tf

mkdir -p ~/labs/ec2-tutorial && cd ~/labs/ec2-tutorial

Save this as main.tf:

terraform {
  required_version = ">= 1.7"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region  = "us-east-1"
  profile = "eddie-lab"

  default_tags {
    tags = {
      Project   = "aws-series-tutorial"
      ManagedBy = "terraform"
    }
  }
}

# Always grab the latest Amazon Linux 2023 AMI for x86_64
data "aws_ami" "al2023" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-*-x86_64"]
  }
  filter {
    name   = "state"
    values = ["available"]
  }
}

# IAM role that EC2 can assume + the SSM managed policy
resource "aws_iam_role" "ssm" {
  name = "ec2-tutorial-ssm-role"
  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Principal = { Service = "ec2.amazonaws.com" }
      Action    = "sts:AssumeRole"
    }]
  })
}

resource "aws_iam_role_policy_attachment" "ssm_core" {
  role       = aws_iam_role.ssm.name
  policy_arn = "arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore"
}

resource "aws_iam_instance_profile" "ssm" {
  name = "ec2-tutorial-ssm-profile"
  role = aws_iam_role.ssm.name
}

# No inbound rules. Outbound default (allow all) is fine — needed for SSM agent to reach the SSM API.
resource "aws_security_group" "tutorial" {
  name        = "aws-series-tutorial-sg"
  description = "EC2 tutorial — no inbound, SSM only"
  # Default VPC; for a real workload you'd reference a vpc_id here
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

resource "aws_instance" "tutorial" {
  ami                    = data.aws_ami.al2023.id
  instance_type          = "t3.micro"
  vpc_security_group_ids = [aws_security_group.tutorial.id]
  iam_instance_profile   = aws_iam_instance_profile.ssm.name

  metadata_options {
    http_tokens   = "required" # IMDSv2 only — defends against SSRF
    http_endpoint = "enabled"
  }

  tags = {
    Name = "aws-series-tutorial"
  }
}

output "instance_id" {
  value       = aws_instance.tutorial.id
  description = "Use this with: aws ssm start-session --target <instance_id>"
}

2. Init, plan, apply

terraform init
terraform plan      # READ THIS. Always read the plan before applying.
terraform apply     # type 'yes' to confirm

Output prints the instance ID. Now go to Connect.

Why this is the production track

Idempotent (apply twice = same result), code-reviewable (your teammate sees the diff), and reversible in one command (terraform destroy). The CLI track is great for learning what's underneath; Terraform is what you'd actually use at work.

06 Connect to your instance (Session Manager)

One command, all three tracks. Replace i-0abc... with your actual instance ID.

aws ssm start-session \
  --target i-0abc123def456 \
  --region us-east-1 \
  --profile eddie-lab

You're now in a shell on the instance, as user ssm-user. Try:

whoami
cat /etc/os-release
sudo dnf check-update
exit       # ends the session
What just happened, security-wise

Your local AWS credentials signed an API call. AWS verified your IAM permissions, opened a websocket through the SSM agent on the instance, and gave you a shell. There was no inbound port on the EC2 instance. The whole thing is logged in CloudTrail (StartSession event) and SSM Session Manager (session-id). You can prove who connected, when, and what they ran. Try doing that with raw SSH.

Troubleshooting

  • "TargetNotConnected" — the SSM agent on the instance isn't reaching the SSM API. Check the IAM role has AmazonSSMManagedInstanceCore. Wait 60 seconds and retry.
  • "plugin not found" — install the Session Manager CLI plugin.
  • The agent times out connecting to the SSM API — the security group's egress rules block it. The default "allow all egress" works; if you tightened egress you need 443 to ssm.<region>.amazonaws.com, ssmmessages, and ec2messages.

07 Cleanup — read this first

Why this matters

A forgotten t3.micro outside free-tier costs ~$7.60/month. A forgotten NAT Gateway costs ~$32/month. A forgotten p3.2xlarge GPU costs $3/hour — $2,200/month. Always destroy before you walk away from the keyboard.

Track C (Terraform) — easiest

cd ~/labs/ec2-tutorial
terraform destroy   # type 'yes' — wipes everything we created

Track B (CLI) — manual

# Terminate the instance (replace with your IDs)
aws ec2 terminate-instances \
  --instance-ids "$INSTANCE_ID" \
  --region us-east-1 \
  --profile eddie-lab

aws ec2 wait instance-terminated \
  --instance-ids "$INSTANCE_ID" \
  --region us-east-1 \
  --profile eddie-lab

# Delete the SG (must wait for instance to terminate first)
aws ec2 delete-security-group \
  --group-id "$SG_ID" \
  --region us-east-1 \
  --profile eddie-lab

# Detach + delete the IAM resources
aws iam remove-role-from-instance-profile \
  --instance-profile-name ec2-tutorial-ssm-profile \
  --role-name ec2-tutorial-ssm-role \
  --profile eddie-lab

aws iam delete-instance-profile \
  --instance-profile-name ec2-tutorial-ssm-profile \
  --profile eddie-lab

aws iam detach-role-policy \
  --role-name ec2-tutorial-ssm-role \
  --policy-arn arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore \
  --profile eddie-lab

aws iam delete-role \
  --role-name ec2-tutorial-ssm-role \
  --profile eddie-lab

Track A (Console)

  1. EC2 → Instances → select your instance → Instance state → Terminate (delete) instance.
  2. Wait for it to show as terminated. Then EC2 → Security Groups → delete aws-series-tutorial-sg.
  3. IAM → Roles → delete ec2-tutorial-ssm-role. The Console deletes the linked instance profile for you.

Verify nothing is left

aws ec2 describe-instances \
  --filters "Name=tag:Project,Values=aws-series-tutorial" \
  --query "Reservations[].Instances[?State.Name!='terminated'].InstanceId" \
  --output text \
  --region us-east-1 \
  --profile eddie-lab
# (empty output = nothing running)

08 Senior-engineer takeaways

What this exercise drilled into you

  • Idempotency — Terraform apply twice = same result. CLI commands aren't idempotent by default; you have to check before creating, or accept duplicates.
  • Blast radius — every resource was tagged Project=aws-series-tutorial so cleanup is grep-able. Tag religiously; future-you will thank you when the bill arrives.
  • Least privilege — AmazonSSMManagedInstanceCore is a managed policy. In production you'd write a custom policy with only the SSM actions you need, and scope it to specific resources.
  • Defense in depth — IMDSv2 + no inbound SG + IAM-authenticated SSM = three independent layers. Removing any one of them is bad. Removing all three is Capital One.
  • Reproducibility — the Terraform code in this tutorial works on your laptop, in CI, and on a fresh AWS account. The Console clicks don't.

Where to take it next

  • Wrap the Terraform in a module. Parameterize instance_type, vpc_id, tags. That's how real codebases organize.
  • Add a user data script so the instance bootstraps itself (install Docker, run a service, etc.).
  • Move from default VPC to a custom VPC with private subnets — that's the next service in this series.
  • Replace the SSM managed policy with a custom least-privilege policy — great AWS Security Specialty exam practice.

Costs at a glance

  • t3.micro: free for 750 hrs/month for 12 months on new accounts. After that ~$7.60/month if running 24/7.
  • EBS: 8 GiB gp3 root volume ~$0.64/month.
  • Data transfer: minimal for SSM. First 100 GB/month egress is free.
  • SSM Session Manager: free.

Pricing reference: AWS EC2 on-demand pricing. Verify before assuming.
