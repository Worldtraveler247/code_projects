Amazon EC2

Elastic Compute Cloud — virtual servers in the sky, billed by the second. The original AWS service (launched 2006) and still the workhorse the entire ecosystem rests on.

01 What is EC2?

EC2 lets you rent a Linux or Windows server, on demand, anywhere from 30 seconds to years, in any of 30+ AWS regions globally — and pay only for what you use, by the second.

Under the hood, every EC2 "instance" is a virtual machine running on top of AWS's Nitro hypervisor on physical hardware in an AWS data center. You pick a size (CPU, memory, network, GPU), an OS image (Amazon Machine Image, or AMI), a network (which VPC and subnet), and a security group (firewall rules) — and a few seconds later you have SSH or RDP into a fresh box.

The objective: turn capital expense into operating expense

Before EC2 (2006), if your startup needed servers you bought servers — racked them, cooled them, paid for the bandwidth, paid an admin to babysit them. If traffic doubled overnight, you were stuck. If traffic dropped, you'd already bought the hardware.

EC2 turned compute into a utility — like electricity. Need 200 servers for 6 hours during a marketing campaign? Spin them up, run the campaign, terminate them. The bill is for 6 hours of 200 servers, not 200 servers forever.

In one sentence

EC2 is renting a computer in the cloud — except the cloud bills you per second, terminates the rental when you tell it to, and lets you make 1,000 copies in parallel.

02 A vibrant example: Netflix on Friday night

It's Friday, 8 p.m. Pacific. Half of North America just got home and queued up the new Stranger Things season. Netflix's traffic graph spikes vertically.

Netflix doesn't own a single server. They run on EC2. Hundreds of thousands of EC2 instances across multiple regions — encoding video, running recommendation models, serving the API that responds to your "play" tap. As traffic spikes, an Auto Scaling group provisions hundreds more EC2 instances in minutes. At 3 a.m. when traffic drops, those instances terminate themselves and the bill stops.

Same hardware, same software, same engineers. The only thing that changed is capacity, in minutes, on demand. That is the EC2 thesis.

03 Who uses EC2 — and why

Almost everyone. EC2 is one of the most widely used cloud services on Earth. Concrete examples (publicly documented):

🎬 Netflix

Streaming, encoding, recommendation ML, A/B testing — all on EC2 across multiple regions. Netflix Tech Blog →

🚀 NASA / JPL

The Mars Curiosity rover landing in 2012 used EC2 to process and stream images to mission control and to the public.

🏠 Airbnb

Migrated from a single server to EC2 in their early years; runs the matching, search, and payment APIs on EC2 + RDS.

📰 Reddit

Was on EC2 for over a decade — the comment-tree workload is famously hard to cache, EC2 + ElastiCache made it work.

🏛️ U.S. Government

AWS GovCloud regions run EC2 for DoD and intel workloads. Same service, FedRAMP High-authorized, isolated network.

🧪 Pharma / Genomics

Pfizer, Moderna, Illumina — burst genome alignment jobs across thousands of GPU/CPU instances, then terminate. The math works only because EC2 is per-second.

Common reasons to reach for EC2

  • Lift-and-shift: existing on-prem application, doesn't want to be rewritten serverless. EC2 looks like a server it already knows how to run on.
  • Custom kernels / GPUs / FPGAs: you need control AWS doesn't expose in higher-level services.
  • Long-running, stateful workloads: databases, message brokers, game servers — anything that doesn't suit Lambda's 15-minute limit.
  • Burst compute: ML training, video transcoding, scientific simulation — pay for what you burn, not what you reserve.
  • Compliance: you need a specific OS image, hardened to a specific benchmark (CIS, STIG), and no PaaS will give you that.

04 Competitors & market share

EC2 has direct rivals in every other major public cloud. Honest framing — AWS is the market leader, but "best" depends on what you're optimizing for.

Provider Compute Service Cloud Market Share Where it wins
AWS EC2 ~30% Most instance types, biggest region footprint, most mature ecosystem, deepest IAM/security primitives.
Microsoft Azure Azure Virtual Machines ~21% Enterprise + Microsoft licensing, Active Directory integration, hybrid (Azure Arc), Windows-heavy shops.
Google Cloud Compute Engine (GCE) ~12% Per-second billing was first here, live migration is excellent, networking and Kubernetes (they invented it).
Alibaba Cloud Elastic Compute Service (ECS) ~4% Dominant in China; the only realistic option there.
Oracle, IBM, others various combined ~33% Specialized: Oracle for Oracle DB workloads, IBM for mainframe-adjacent.

Market share figures: Synergy Research Group, Q4 2024. These shift quarterly — verify before quoting.

The honest "why AWS"

AWS isn't universally better — it's the most likely default. Reasons: biggest catalog of instance types and adjacent services, the deepest set of compliance certifications, the most engineers in the wild who already know it, and the largest pool of pre-existing infrastructure-as-code modules. For most teams that translates to lower friction and a shallower learning curve per engineer hired. That's the real moat — not raw technical superiority.

Where AWS doesn't always win

  • Kubernetes-first workloads: GKE on Google Cloud is generally considered the best-in-class managed Kubernetes.
  • Microsoft-shop enterprises: Azure's licensing bundle (Office 365 + AD + VMs) makes it cheaper than AWS for them.
  • Egress costs: AWS data-egress pricing is famously expensive. CDNs and competitors with cheaper egress (e.g., Cloudflare R2) can dramatically beat AWS on bandwidth-heavy workloads.
  • China: AWS exists in China but Alibaba is dominant due to regulation and latency.

05 Who actually runs it for you?

EC2's "shared responsibility" line: AWS runs the hardware, hypervisor, and physical data center. You run the OS, the patches, the application, the network rules, and the data. But who, on your side, does that?

👤 Self-managed (startup / solo)

You launch instances, SSH in, install software, patch the OS, read CloudWatch dashboards yourself. Minimum cost, maximum 2 a.m. pages. Most learning happens here.

🏢 Self-managed (small team)

A platform / DevOps engineer wraps EC2 in Terraform + Ansible + CI/CD. Developers don't touch the Console; they merge a PR and infrastructure changes.

🛠️ AWS Professional Services

AWS-paid solutions architects help large customers design and migrate — usually a paid engagement, weeks to months. They don't run your day-to-day ops.

📡 AWS Managed Services (AMS)

A paid AWS service for enterprise customers — AWS handles patching, monitoring, change management. You give up some flexibility for hands-off ops.

🤝 Third-party MSP

Companies like Mission, Cloudreach, Rackspace specialize in running AWS for you. Common for mid-market companies without a platform team.

🤖 Higher-level abstractions

Many teams sidestep EC2 entirely — ECS Fargate, Lambda, App Runner manage instances for you. EC2 is the foundation; you may never touch it directly.

⚠️ Startup gotcha

First-time EC2 users frequently leave instances running by accident — "I'll terminate it later" becomes a $300 surprise bill the next month. Set a billing alarm before your first launch. We'll wire one up in the tutorial.

06 Security mindset

Most "AWS breaches" you read about aren't AWS being broken — they're EC2/S3/IAM being misconfigured by the customer. The 2019 Capital One breach (100M records) was an SSRF + over-privileged EC2 instance role. Read it twice; it's the most instructive case study in cloud security history. Capital One disclosure →

Five things to internalize before your first production EC2

  • Use IAM roles, not access keys. An EC2 instance attached to a role gets short-lived credentials automatically. Long-lived keys end up in git.
  • Use Session Manager, not SSH. AWS Systems Manager Session Manager gives you a shell over IAM with full audit logs — no port 22 open to the world, no SSH keys to lose.
  • Security Groups are stateful firewalls. Open ports to specific source SGs or CIDRs, not 0.0.0.0/0. "Allow from anywhere" is the leading cause of EC2 compromise.
  • Patch with SSM Patch Manager. Automated, scheduled, auditable. Beats "I'll dnf update sometime."
  • Enable IMDSv2. Defense against the SSRF class of attacks that took down Capital One. As of 2024 new instances default to IMDSv2; older AMIs may not.
For your career

If you're targeting cloud security roles, EC2 is where you'll spend a huge chunk of time auditing. Get fluent with Security Groups, IAM roles, IMDSv2, and SSM — these come up in every AWS Security Specialty exam question and every real incident response.
