Amazon VPC

Virtual Private Cloud — your own software-defined network inside AWS. Every EC2 instance, every Lambda function, every database lives in a VPC. It's the most important AWS service nobody talks about.

01 What is VPC?

A VPC is a logically isolated, software-defined network inside an AWS region. You define the IP address range (CIDR block), carve it into subnets across Availability Zones, control routing, and decide what reaches the internet — and what doesn't.

Before VPC (it became the default in 2013), AWS resources lived on a flat shared network ("EC2-Classic"). Today every region in your account starts with a default VPC, and you can create more for isolation, blast-radius control, or compliance separation. The default VPC is fine for labs; production almost never uses it.

The components you actually need to know

  • CIDR block — the IP range, e.g. 10.0.0.0/16 (65,536 addresses). Pick this carefully — it's hard to change later, and it must not overlap with networks you'll peer to.
  • Subnets — slices of the CIDR, each pinned to one Availability Zone. Public subnets have a route to the internet via an Internet Gateway. Private subnets don't.
  • Route tables — per-subnet rules saying "traffic for X goes to Y." This is where most "I can't reach the internet" debugging ends up.
  • Internet Gateway (IGW) — one per VPC, attaches the VPC to the public internet. Public subnets route through it.
  • NAT Gateway — lets private subnets reach the internet for updates/APIs without being reachable from the internet. ~$32/month idle plus data transfer. Famous bill-shocker.
  • Security Groups — stateful firewalls attached to ENIs (network interfaces). Allow-only; default deny.
  • Network ACLs (NACLs) — stateless firewalls at the subnet boundary. Useful for explicit denies.
  • VPC Endpoints — private connections to AWS services (S3, DynamoDB, SSM, etc.) without going to the internet. Critical for security and for cutting NAT costs.
  • VPC Flow Logs — packet metadata for every flow. The audit log of your network.
In one sentence

VPC is your data center, redrawn as code. You get the network primitives (subnets, routes, firewalls, gateways) without the cabling — and you version-control the whole topology in Terraform.

02 A vibrant example: a fintech that handles credit cards

Imagine a small fintech startup. They take credit card payments, which means they're in scope for PCI DSS — the cardholder-data compliance standard. PCI auditors are strict about segmentation: cardholder data systems must be isolated from everything else.

Without VPC, that's a hardware problem — separate VLANs, separate firewalls, cabling diagrams. With VPC, it's a Terraform pull request:

  • Three subnets across two AZs: web-public, app-private, db-isolated.
  • web-public has a route to the IGW. The Application Load Balancer lives there. Inbound: 443 from the internet only.
  • app-private has a route to the NAT Gateway (so it can pull updates) but no inbound from the internet. EC2 / Fargate tasks live here. Inbound: 8080 from the LB security group only.
  • db-isolated has no internet route at all. PostgreSQL on RDS. Inbound: 5432 from the app security group only. Outbound: nothing.
  • VPC Endpoints for S3, KMS, and Secrets Manager so the app can reach those services without ever touching the public internet.
  • VPC Flow Logs to CloudWatch — every packet metadata is logged, queryable for incident response.

Auditor walks in. "Show me your segmentation." Engineer opens vpc.tf. Audit done in twenty minutes instead of twenty days. That's the VPC value proposition.

03 Who uses VPC — and why

Everyone with AWS resources, whether they know it or not. Every EC2, RDS, ElastiCache, Lambda-with-VPC, EKS pod, ECS task — they all live inside a VPC. The question isn't "do I use VPC" but "did I take the defaults or design my own."

🏦 Regulated industries

Banks, healthcare, fintech use custom VPCs to enforce segmentation for PCI / HIPAA / SOC 2. Subnets become the auditor-readable boundary.

🏢 Enterprises with hybrid cloud

Connect VPCs to on-prem data centers via Site-to-Site VPN or Direct Connect. The VPC is the AWS-side endpoint of the corporate network.

🌐 Multi-account organizations

AWS Organizations + Transit Gateway connects dozens of VPCs across accounts. The VPC is the unit of network ownership per team / per environment.

🛡️ Security-first startups

Default VPCs are too permissive. Even small teams build custom VPCs with private-by-default subnets and explicit egress rules.

🎮 Multi-region gaming / streaming

One VPC per region (VPCs are regional), peered or connected via Transit Gateway. Players hit the closest region's VPC.

🎓 Learners (you, today)

The default VPC is free-tier friendly and works fine for labs. Move to a custom VPC the moment your data is real.

Why design a custom VPC instead of using the default?

  • Default VPC has 5 public subnets and an open route to the internet. No private/isolated tier exists out of the box. That's wrong for production.
  • Default Security Group allows all egress. A compromised host can phone home anywhere.
  • You can't change the default VPC's CIDR. If you ever want to peer with another network, that 172.31.0.0/16 might overlap.
  • Auditability. A custom VPC's Terraform file is your network diagram. The default VPC is undocumented.

04 Competitors & market share

Every major cloud has a VPC equivalent. The concept is the same; the model differs in interesting ways — and for once, AWS doesn't clearly win.

Provider Service Scope Where it wins / loses
AWS VPC Regional Most mature, most adjacent services. Loss: regional-only — multi-region needs Transit Gateway / peering.
Microsoft Azure Virtual Network (VNet) Regional Service tags simplify NSG rules ("AllowFromAzureSql"). Loss: NSG model is less granular than AWS Security Groups.
Google Cloud VPC Global Best-in-class: a single GCP VPC spans all regions — no peering for cross-region traffic. Routing is simpler. Networking is widely considered Google's strongest area.
Alibaba Cloud VPC Regional Concept matches AWS. Dominant in China.
Oracle Cloud VCN (Virtual Cloud Network) Regional Free internal egress within a region. Loss: smaller ecosystem.
The honest take

If you start over today, GCP's global VPC is genuinely the cleaner model. AWS's regional VPC + Transit Gateway is more complex and more expensive at multi-region scale. AWS still wins overall because of ecosystem maturity and the surrounding services (security groups, VPC endpoints, the IAM model) — but the network primitive itself isn't where AWS leads. Worth knowing for interviews and for designing your own multi-cloud architecture.

05 Who actually runs it for you?

VPC is unusual: it's configuration, not workload. There's no "VPC instance" to patch, no service to keep running. You design it, deploy it, and largely leave it alone — but the design decisions are permanent.

👤 Self-managed (solo / startup)

You design your CIDR plan, write the Terraform, and own every Security Group. CIDRs picked at the start are nearly unchangeable — get it right once.

🛠️ Platform / Network team

At any company past 50 engineers, a platform team owns the VPC topology, peering, and Transit Gateway. Application teams request subnets / SGs via PR.

🏢 AWS Landing Zone / Control Tower

Large enterprises use AWS Control Tower to provision pre-baked multi-account VPC architectures with guardrails. Removes a class of "first-month" mistakes.

📡 Managed networking products

Aviatrix, Cisco Cloud ACI, Palo Alto Cloud NGFW sit on top of VPCs to give a multi-cloud / on-prem-style network experience.

🤝 Third-party MSP

If you've outsourced AWS ops to Mission / Cloudreach / Rackspace, they design the VPC. You should still review the Terraform — VPC mistakes are permanent.

🤖 Higher-level abstractions

VPC Lattice (newer AWS service) abstracts service-to-service connectivity above the VPC layer. App Runner / Lambda hide VPCs entirely. The trend is toward less hand-built networking.

⚠️ Permanent decisions

Two things are extremely hard to change later: your CIDR block and your account-VPC topology. Pick the CIDR with future peering in mind (avoid 10.0.0.0/16 — it's everyone's default and will collide). Decide early whether you want one VPC or many. Migrating workloads between VPCs after the fact is painful.

06 Security mindset

VPC is where most cloud security actually happens. Misconfigured IAM gets the headlines, but misconfigured networking is what turns a small compromise into a big one.

Security Groups vs Network ACLs — the one chart everyone gets wrong

  Security Group Network ACL
Scope Attached to ENIs (instance-level) Attached to subnets
State Stateful — return traffic auto-allowed Stateless — must allow both directions
Rules Allow only (no explicit deny) Allow and deny, evaluated in order
Default Default SG: deny inbound, allow all outbound Default NACL: allow all both ways
Use it for Almost everything. Reach for SGs first. Explicit blocks (e.g., block a malicious CIDR temporarily)

Five things to internalize

  • Default SG allows all outbound. A compromised host inherits this. Lock egress down — only allow the destinations the workload actually needs.
  • Use VPC Endpoints for AWS services. S3, DynamoDB, SSM, Secrets Manager — going through endpoints keeps traffic on the AWS network, audits via Endpoint Policies, and saves NAT Gateway $.
  • Enable VPC Flow Logs. They're cheap, queryable in CloudWatch Logs Insights or Athena, and they're how you'll find lateral movement during an incident response.
  • Private subnets + SSM Session Manager > bastion hosts. Don't put EC2 in a public subnet "for SSH access." See the EC2 page for the modern pattern.
  • Watch the NAT bill. A NAT Gateway is ~$32/month idle plus $0.045/GB processed. A misbehaving service can rack up thousands. NAT instances or VPC endpoints are cheaper alternatives in some cases.
For your career

VPC questions are heavy on the AWS Solutions Architect and especially the Security Specialty exam. Get fluent with: subnet routing, NACLs vs SGs, VPC Endpoints (gateway vs interface), Flow Logs, Transit Gateway use cases, and the cost model of NAT. These come up in every AWS networking interview question.
