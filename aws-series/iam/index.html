AWS IAM

Identity and Access Management — the blast-radius control plane for your AWS account. The service that decides who can do what to which resource, and the one most likely to be quoted in a breach post-mortem when it's wrong.

01 What is IAM?

IAM is the service that answers "is this caller allowed to perform this action on this resource?" for every API call inside your AWS account — from s3:GetObject on a single file, to iam:DeleteUser on the human who created the account.

Every AWS API call passes through IAM. AWS receives a request, looks up the caller's identity (a user, a role, or a service), evaluates every policy attached to that identity and to the target resource, and either allows the call or returns AccessDenied. There is no AWS API call that bypasses this check — not from the Console, not from the CLI, not from inside an EC2 instance.

The four primitives

  • Users — long-lived identities for humans or legacy automation. Have passwords and/or access keys. Strongly discouraged in 2026 for humans (use SSO/Identity Center) and for workloads (use roles).
  • Roles — identities you assume. Issue short-lived credentials (15 min – 12 hr) via STS. The right answer for EC2 instances, Lambda functions, cross-account access, and federated humans.
  • Policies — JSON documents that describe what's allowed or denied. Attached to identities (identity-based) or to resources (resource-based, e.g. an S3 bucket policy).
  • Groups — bundles of users that share the same policies. Convenient for human management; irrelevant for workload identity (which uses roles).
In one sentence

IAM is "who can do what" expressed as JSON, evaluated on every API call, with deny winning over allow.

02 A vibrant example: Capital One, 2019

A single misconfigured IAM permission on an EC2 instance role exposed 106 million customer records — the most-cited cloud-security case study in modern history, and an IAM lesson at every layer.

The attacker exploited an SSRF vulnerability in a Capital One web app to make the EC2 instance call its own Instance Metadata Service and hand over its IAM role's temporary credentials. Those credentials were then used against S3 — and because the role had s3:ListAllMyBuckets + s3:GetObject on production buckets it had no business reading, the attacker walked out with credit applications, SSNs, and bank account numbers.

Three IAM-shaped failures stacked: (1) the role had way more permissions than the workload needed; (2) IMDSv1 was enabled, letting the SSRF reach the metadata endpoint; (3) no condition keys restricted which buckets the role could read. Any one of those fixed in isolation would have stopped the breach. That's IAM in production. Capital One disclosure →

03 Who uses IAM — and why

Everyone with an AWS account uses IAM, whether they realize it or not. The interesting question is how seriously different organizations take it. A few publicly documented examples:

🏛️ U.S. Department of Defense

Runs JWCC and earlier JEDI/JWICS workloads in AWS GovCloud. IAM policies are the security boundary between unclassified, secret, and TS workloads — audited continuously by independent assessors.

🏦 Capital One (post-2019)

Rewrote their entire IAM posture after the breach: SCPs at the org level, permission boundaries on every role, automated continuous-compliance scanners. Capital One Tech →

🎬 Netflix

Open-sourced Repokid and Aardvark to right-size IAM permissions automatically — strip privileges that haven't been used in 90 days. Netflix Tech Blog →

🏢 Anduril, Palantir, defense-tech

Cleared workloads use IAM Identity Center federated to the company's ID provider, with roles assumed via short-lived SAML/OIDC tokens. No long-lived access keys touch human laptops.

🚀 Startups (the cautionary tale)

The first 3 engineers all use the root account. Then a contractor commits AWS keys to GitHub. The bill is $40,000 of crypto-mining the next morning. IAM exists to prevent this exact story.

🛂 Federal agencies (FedRAMP)

FedRAMP Moderate and High require detailed IAM controls — least privilege, MFA, audited via CloudTrail, role-based access. IAM is the spine of every FedRAMP authorization.

Common reasons IAM gets called out specifically

  • Compliance frameworks — SOC 2, PCI-DSS, HIPAA, FedRAMP all have control families that map directly to IAM policies, MFA, and access reviews.
  • Multi-account architectures — AWS Organizations + Service Control Policies (SCPs) + cross-account IAM roles are how big companies isolate prod from dev from sandbox.
  • Workload identity — every modern Lambda, ECS task, EC2 instance, EKS pod gets a role. Long-lived keys are 2010-thinking.
  • Federation — IAM Identity Center (formerly AWS SSO) lets your Okta / Entra ID / Google Workspace handle authentication, with IAM handling authorization once they land.
  • Incident response — when something is wrong, "what permissions did this principal actually have, and what did it use?" is always the first IAM question. Access Analyzer + CloudTrail answer it.

04 Competitors & market share

IAM isn't a standalone product — it's bundled with the cloud. So the comparison is really "how each cloud vendor models identity and authorization." Honest framing: the model choices are different enough that switching is hard, but the underlying problem is the same.

The honest "why AWS IAM"

IAM has the most expressive policy language of the three majors — resource-based policies, condition keys, organization-wide guardrails (SCPs), permission boundaries, session policies. That expressiveness is also why it's the easiest to get wrong. Azure's RBAC and GCP IAM are both simpler; whether that's better depends on whether you need the precision.

Where AWS IAM doesn't always win

  • Human identity UX: Entra ID's conditional access, device trust, and risk-based MFA are noticeably ahead of IAM Identity Center.
  • Mental model overhead: AWS has users, groups, roles, identity-based policies, resource-based policies, permission boundaries, SCPs, session policies, and ACLs. GCP's "principal + role on resource hierarchy" is much smaller.
  • Cross-cloud: if you're running on AWS + Azure + GCP, Vault or a single IdP is friendlier than maintaining three native IAM stacks.

05 Who actually runs it for you?

IAM's "shared responsibility" line: AWS runs the IAM service itself (control plane, policy evaluation, STS). You write every policy, every role's trust relationship, and every permission boundary. Misconfigurations are entirely on the customer.

👤 Self-managed (startup / solo)

You're the IAM admin and the developer and the on-call. Easy to over-grant "to make it work" and never circle back. Most learning happens here; most over-privileged roles also start here.

🏢 Cloud platform team

A central platform team owns the IAM baseline (SCPs, permission boundaries, role templates). Developers request roles via PR or a self-service portal — never write policies from scratch.

🛡️ Cloud security engineering

Distinct from platform. Owns guardrails, continuous compliance scanning (Access Analyzer, Security Hub), and incident response when an IAM credential leaks. Often the team that hires for this exact role.

🤝 Third-party MSP

Mid-market companies often outsource IAM hygiene to MSPs (Mission, Cloudreach, Rackspace, etc.). The MSP runs the IdP federation, role templates, and audits.

🤖 Automated tooling

Repokid (Netflix), Cloudsplaining (Salesforce), Pacu, Prowler — open-source tools that audit IAM state continuously. Increasingly run inside CI to block over-privileged policies before they merge.

🛂 Compliance / GRC

FedRAMP / SOC 2 / PCI assessors don't manage IAM, but they read it. Quarterly access reviews are usually their deliverable — the IAM team just exports the data.

⚠️ Startup gotcha

First-time AWS users often grant AdministratorAccess to "make it work for now" and never revisit. That role is the entire account. If it's attached to an EC2 instance whose web app has an SSRF, you're Capital One. Treat AdministratorAccess the way you'd treat sudo -i: rare, audited, never automatic.

06 Security mindset

IAM is the security mindset. Every other AWS service's security story bottoms out in "what permissions does the calling identity have." Get IAM right and most other services get easier; get it wrong and nothing else matters.

Five things to internalize before you write your first policy

  • Least privilege isn't a slogan, it's a measurable property. Use IAM Access Analyzer and CloudTrail-derived policy generation to check what permissions were actually used versus granted. Strip the rest.
  • Roles for workloads, federation for humans. Long-lived access keys belong in a 2014 post-mortem, not in 2026 production. EC2 / Lambda / ECS get instance roles; humans get Identity Center + SSO.
  • Default deny, explicit allow, deny wins. An IAM policy without a matching Allow denies. An Allow at any level is overridden by a Deny at any level. SCPs and permission boundaries deny — that's their job.
  • Use conditions to scope. aws:SourceIp, aws:RequestTag, aws:PrincipalOrgID, kms:ViaService — condition keys turn "allow this action" into "allow this action only when X." Skip them and you've granted the unconditional version.
  • Audit continuously, review periodically. CloudTrail logs every IAM call. Access Analyzer flags policies granting external access. Quarterly access reviews catch the stale grants no automation will.
For your career

If you're targeting cloud security roles, IAM is the single highest-leverage skill on the AWS Security Specialty exam and in every real engagement. Every breach story bottoms out in "the policy was wrong" or "the credential leaked." Become fluent in the policy language — actions, resources, conditions, principals — and you're already more useful than half the field.
