IAM: Build a Least-Privilege Role

Three tracks, same lab. We build a role for a Lambda function that can read one specific S3 prefix and nothing else — then prove it using IAM Access Analyzer and CloudTrail.

What's in this tutorial

  1. Prerequisites & safety net
  2. What we'll build
  3. Track A — Console (click-by-click)
  4. Track B — AWS CLI v2
  5. Track C — Terraform
  6. Verify least privilege (Access Analyzer + simulator)
  7. Cleanup — read this first
  8. Senior-engineer takeaways

01 Prerequisites & safety net

You'll need

  • An AWS account (everything in this tutorial is free).
  • An IAM user or Identity Center role with permission to create IAM roles, policies, and S3 buckets. Don't use the root account.
  • AWS CLI v2 installed (aws --version should print 2.x).
  • Terraform 1.7+ for Track C.

Configure a named CLI profile (one-time setup)

Don't dump credentials into env vars; use named profiles in ~/.aws/credentials. Replace eddie-lab with whatever you want to call this profile.

aws configure --profile eddie-lab
# AWS Access Key ID [None]: AKIA...
# AWS Secret Access Key [None]: ...
# Default region name [None]: us-east-1
# Default output format [None]: json

# Verify
aws sts get-caller-identity --profile eddie-lab

Every CLI command in this tutorial uses --profile eddie-lab --region us-east-1. Adjust to your profile/region.

⚠️ A note about the principle of least privilege

The temptation is always to attach AdministratorAccess, get the lab working, and "tighten it later." Resist. The whole point of this tutorial is to feel the friction of writing a tight policy from scratch — that friction is the muscle memory that prevents the next Capital One.

02 What we'll build

A role that an AWS Lambda function can assume, with permission to:

  • s3:GetObject on objects under exactly one prefix in exactly one bucket.
  • Write its own logs to CloudWatch (the basic Lambda execution-role permissions).
  • Nothing else. No s3:ListAllMyBuckets, no s3:GetObject on other buckets, no iam:*, no ec2:*.

We'll then prove the policy is tight by running it through the IAM policy simulator and Access Analyzer — the same tools auditors use.

Why this lab

"Write a least-privilege Lambda role" is the AWS Security Specialty exam in microcosm. Every condition key, every Resource ARN, every Effect matters. After this lab you'll know exactly what's in AmazonS3ReadOnlyAccess — and why you'd never attach it to a real workload.

Track A Console — click-by-click

Best for the first time, when you want to see the policy editor visually.

  1. Sign in at console.aws.amazon.com as your IAM user (not root). Pick region US East (N. Virginia) us-east-1 top right.
  2. Create a test S3 bucket. Go to S3 → Create bucket.
    • Name: iam-tutorial-<your-name>-<random-4-chars> (S3 names are global).
    • Block all public access: checked (default).
    • Create bucket. Then upload any small file under prefix readable/ (e.g. readable/hello.txt) and another under secret/ (e.g. secret/keep-out.txt).
  3. Create the IAM policy. Go to IAM → Policies → Create policy. Switch to the JSON tab and paste: 
    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadOnlyOneBucketPrefix",
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::iam-tutorial-<your-name>-<random>/readable/*"
    }
  ]
}
    Name it iam-tutorial-s3-readable-prefix. Create policy.
  4. Create the role. Go to IAM → Roles → Create role.
    • Trusted entity: AWS service → use case Lambda.
    • Permissions: attach AWSLambdaBasicExecutionRole (CloudWatch logs) and your iam-tutorial-s3-readable-prefix policy.
    • Name: iam-tutorial-lambda-role.
    • Create role.
  5. Sanity check. Open the role, click Permissions → Simulate. Test:
    • s3:GetObject on arn:aws:s3:::iam-tutorial-…/readable/hello.txt → allowed.
    • s3:GetObject on arn:aws:s3:::iam-tutorial-…/secret/keep-out.txt → implicit deny.
    • s3:DeleteObject on either → implicit deny.

That's a least-privilege Lambda role. Skip ahead to Verify for the audit step.

Track B AWS CLI v2

Same lab, all from your terminal. Worth doing once — you'll learn what the Console buttons actually call.

1. Create a test bucket and seed it

BUCKET="iam-tutorial-$(whoami)-$(openssl rand -hex 2)"

aws s3api create-bucket \
  --bucket "$BUCKET" \
  --region us-east-1 \
  --profile eddie-lab

# Two objects under two different prefixes
echo "hello" | aws s3 cp - "s3://$BUCKET/readable/hello.txt" --profile eddie-lab
echo "secret" | aws s3 cp - "s3://$BUCKET/secret/keep-out.txt" --profile eddie-lab

echo "Bucket: $BUCKET"

2. Write the policy and trust documents

# The policy that actually grants access
cat > /tmp/iam-tutorial-policy.json <<EOF
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadOnlyOneBucketPrefix",
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::${BUCKET}/readable/*"
    }
  ]
}
EOF

# Trust policy (lets Lambda assume the role)
cat > /tmp/iam-tutorial-trust.json <<'EOF'
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "Service": "lambda.amazonaws.com" },
    "Action": "sts:AssumeRole"
  }]
}
EOF

3. Create the policy + role

# Create the customer-managed policy
POLICY_ARN=$(aws iam create-policy \
  --policy-name iam-tutorial-s3-readable-prefix \
  --policy-document file:///tmp/iam-tutorial-policy.json \
  --query 'Policy.Arn' \
  --output text \
  --profile eddie-lab)

# Create the role with the Lambda trust policy
aws iam create-role \
  --role-name iam-tutorial-lambda-role \
  --assume-role-policy-document file:///tmp/iam-tutorial-trust.json \
  --profile eddie-lab

# Attach our custom policy AND the AWS-managed basic-execution policy
aws iam attach-role-policy \
  --role-name iam-tutorial-lambda-role \
  --policy-arn "$POLICY_ARN" \
  --profile eddie-lab

aws iam attach-role-policy \
  --role-name iam-tutorial-lambda-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole \
  --profile eddie-lab

4. Verify with the policy simulator

ROLE_ARN=$(aws iam get-role \
  --role-name iam-tutorial-lambda-role \
  --query 'Role.Arn' \
  --output text \
  --profile eddie-lab)

# Should be allowed
aws iam simulate-principal-policy \
  --policy-source-arn "$ROLE_ARN" \
  --action-names s3:GetObject \
  --resource-arns "arn:aws:s3:::${BUCKET}/readable/hello.txt" \
  --query 'EvaluationResults[0].EvalDecision' \
  --output text \
  --profile eddie-lab

# Should be implicitly denied
aws iam simulate-principal-policy \
  --policy-source-arn "$ROLE_ARN" \
  --action-names s3:GetObject \
  --resource-arns "arn:aws:s3:::${BUCKET}/secret/keep-out.txt" \
  --query 'EvaluationResults[0].EvalDecision' \
  --output text \
  --profile eddie-lab

Now skip to Verify for the Access Analyzer step.

Track C Terraform

The production track. Idempotent, code-reviewable, destroy-safe.

1. Create a working directory and main.tf

mkdir -p ~/labs/iam-tutorial && cd ~/labs/iam-tutorial

Save this as main.tf:

terraform {
  required_version = ">= 1.7"
  required_providers {
    aws    = { source = "hashicorp/aws", version = "~> 5.0" }
    random = { source = "hashicorp/random", version = "~> 3.5" }
  }
}

provider "aws" {
  region  = "us-east-1"
  profile = "eddie-lab"

  default_tags {
    tags = {
      Project   = "iam-tutorial"
      ManagedBy = "terraform"
    }
  }
}

# Globally-unique bucket name suffix
resource "random_id" "suffix" {
  byte_length = 2
}

resource "aws_s3_bucket" "lab" {
  bucket        = "iam-tutorial-${random_id.suffix.hex}"
  force_destroy = true # only because this is a lab
}

# Seed two objects: one we'll grant access to, one we won't
resource "aws_s3_object" "readable" {
  bucket  = aws_s3_bucket.lab.id
  key     = "readable/hello.txt"
  content = "hello\n"
}

resource "aws_s3_object" "secret" {
  bucket  = aws_s3_bucket.lab.id
  key     = "secret/keep-out.txt"
  content = "secret\n"
}

# Trust policy — only Lambda may assume this role
data "aws_iam_policy_document" "lambda_trust" {
  statement {
    effect  = "Allow"
    actions = ["sts:AssumeRole"]
    principals {
      type        = "Service"
      identifiers = ["lambda.amazonaws.com"]
    }
  }
}

# Permissions policy — exactly one action, on exactly one prefix
data "aws_iam_policy_document" "s3_readable_prefix" {
  statement {
    sid       = "ReadOnlyOneBucketPrefix"
    effect    = "Allow"
    actions   = ["s3:GetObject"]
    resources = ["${aws_s3_bucket.lab.arn}/readable/*"]
  }
}

resource "aws_iam_policy" "s3_readable_prefix" {
  name   = "iam-tutorial-s3-readable-prefix"
  policy = data.aws_iam_policy_document.s3_readable_prefix.json
}

resource "aws_iam_role" "lambda" {
  name               = "iam-tutorial-lambda-role"
  assume_role_policy = data.aws_iam_policy_document.lambda_trust.json
}

resource "aws_iam_role_policy_attachment" "basic_execution" {
  role       = aws_iam_role.lambda.name
  policy_arn = "arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole"
}

resource "aws_iam_role_policy_attachment" "s3_readable_prefix" {
  role       = aws_iam_role.lambda.name
  policy_arn = aws_iam_policy.s3_readable_prefix.arn
}

output "role_arn" {
  value       = aws_iam_role.lambda.arn
  description = "Plug into a Lambda function or aws iam simulate-principal-policy"
}

output "bucket_name" {
  value = aws_s3_bucket.lab.id
}

2. Init, plan, apply

terraform init
terraform plan      # READ THIS. Always read the plan before applying.
terraform apply     # type 'yes' to confirm

Outputs print the role ARN and bucket name. Now go to Verify.

Why this is the production track

Idempotent (apply twice = same result), code-reviewable (your teammate sees the diff), and reversible in one command (terraform destroy). The CLI track is great for learning what's underneath; Terraform is what you'd actually use at work.

06 Verify least privilege (Access Analyzer + simulator)

The role exists. The policy looks right. Prove it's tight using the same tools an auditor would.

Policy simulator — exhaustive what-if

For each (action, resource) pair, ask AWS what the role's policies actually decide.

ROLE_ARN="arn:aws:iam::<ACCOUNT_ID>:role/iam-tutorial-lambda-role"
BUCKET="iam-tutorial-..."

# Allowed
aws iam simulate-principal-policy \
  --policy-source-arn "$ROLE_ARN" \
  --action-names s3:GetObject \
  --resource-arns "arn:aws:s3:::${BUCKET}/readable/hello.txt" \
  --profile eddie-lab \
  --query 'EvaluationResults[0].[EvalActionName,EvalDecision]' \
  --output text

# Implicit deny — wrong prefix
aws iam simulate-principal-policy \
  --policy-source-arn "$ROLE_ARN" \
  --action-names s3:GetObject \
  --resource-arns "arn:aws:s3:::${BUCKET}/secret/keep-out.txt" \
  --profile eddie-lab \
  --query 'EvaluationResults[0].[EvalActionName,EvalDecision]' \
  --output text

# Implicit deny — wrong action
aws iam simulate-principal-policy \
  --policy-source-arn "$ROLE_ARN" \
  --action-names s3:DeleteObject s3:PutObject s3:ListBucket \
  --resource-arns "arn:aws:s3:::${BUCKET}/readable/hello.txt" \
  --profile eddie-lab \
  --query 'EvaluationResults[*].[EvalActionName,EvalDecision]' \
  --output table

Access Analyzer — does anything outside our account get in?

Access Analyzer is the AWS-native tool that detects external trust paths and unused permissions.

# One-time: create the analyzer in your account (free)
aws accessanalyzer create-analyzer \
  --analyzer-name iam-tutorial-analyzer \
  --type ACCOUNT \
  --region us-east-1 \
  --profile eddie-lab

# List findings — should be empty for the role we built (no external access)
aws accessanalyzer list-findings \
  --analyzer-arn "$(aws accessanalyzer list-analyzers --query 'analyzers[0].arn' --output text --profile eddie-lab)" \
  --filter '{"resourceType":{"eq":["AWS::IAM::Role"]}}' \
  --region us-east-1 \
  --profile eddie-lab

You can also generate a policy from CloudTrail history: IAM → Roles → iam-tutorial-lambda-role → Generate policy based on CloudTrail events. After the role is used in production for a while, this tells you exactly which actions it actually called — strip everything else.

The audit pattern

Simulator answers "what could this role do?" Access Analyzer answers "is anything outside my account / org reachable through this role?" CloudTrail-derived policy generation answers "what did this role actually need over the last 90 days?" Use all three.

07 Cleanup — read this first

Why this matters

IAM resources don't cost money — but a forgotten role with broad permissions is a latent vulnerability. Delete labs when you're done. Future-you will not remember which one of 14 tutorial-* roles was the safe one.

Track C (Terraform) — easiest

cd ~/labs/iam-tutorial
terraform destroy   # type 'yes' — wipes everything we created

Track B (CLI) — manual

# Detach policies from the role
aws iam detach-role-policy \
  --role-name iam-tutorial-lambda-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole \
  --profile eddie-lab

aws iam detach-role-policy \
  --role-name iam-tutorial-lambda-role \
  --policy-arn "$POLICY_ARN" \
  --profile eddie-lab

# Delete role + custom policy
aws iam delete-role --role-name iam-tutorial-lambda-role --profile eddie-lab
aws iam delete-policy --policy-arn "$POLICY_ARN" --profile eddie-lab

# Empty + delete bucket
aws s3 rm "s3://$BUCKET" --recursive --profile eddie-lab
aws s3api delete-bucket --bucket "$BUCKET" --region us-east-1 --profile eddie-lab

# Delete the analyzer (free, but keep things tidy)
ANALYZER_ARN=$(aws accessanalyzer list-analyzers --query 'analyzers[0].arn' --output text --profile eddie-lab)
aws accessanalyzer delete-analyzer --analyzer-name iam-tutorial-analyzer --profile eddie-lab

Track A (Console)

  1. IAM → Roles → delete iam-tutorial-lambda-role. The Console offers to detach policies for you.
  2. IAM → Policies → delete iam-tutorial-s3-readable-prefix.
  3. S3 → empty + delete the bucket you created.
  4. IAM Access Analyzer → Settings → delete the analyzer. Free either way; tidy is better.

Verify nothing is left

aws iam list-roles \
  --query "Roles[?contains(RoleName, 'iam-tutorial')].RoleName" \
  --profile eddie-lab
# (empty output = nothing left)

aws iam list-policies --scope Local \
  --query "Policies[?contains(PolicyName, 'iam-tutorial')].PolicyName" \
  --profile eddie-lab
# (empty output = nothing left)

08 Senior-engineer takeaways

What this exercise drilled into you

  • Resource scoping is the policy. The difference between safe and Capital One is arn:aws:s3:::bucket/prefix/* versus *. Get fluent reading ARNs.
  • Implicit deny is the default. Anything not explicitly allowed is denied. You don't need to write deny statements for actions you didn't grant — leave them out.
  • Trust policies and permission policies do different jobs. Trust policy = "who can become this role." Permission policy = "what this role can do once they have it." Both wrong = breach.
  • AWS-managed policies are starting points, not endings. AmazonS3ReadOnlyAccess grants read on every bucket. For real workloads, write a custom policy scoped to one bucket and one prefix.
  • Verify, don't trust. Simulator + Access Analyzer + CloudTrail-derived generation are the audit pyramid. Use them on every non-trivial policy.

Where to take it next

  • Add a condition key — restrict the role to your org (aws:PrincipalOrgID) or a source IP range (aws:SourceIp).
  • Build a permission boundary — a policy that caps what the role can ever do, even if a future policy attachment grants more. Standard guardrail in regulated environments.
  • Add an SCP at the org level denying iam:CreateUser account-wide — force everyone onto Identity Center + roles.
  • Take the AWS Security Specialty (SCS-C02) exam. The IAM domain is ~30% of it; this lab is the muscle memory.

Costs at a glance

  • IAM users, roles, policies: free.
  • IAM Access Analyzer (account-level findings): free. The unused-access tier costs ~$0.20/role/month — we didn't enable it here.
  • S3 bucket: free under 5GB / 12 months on new accounts. Two tiny objects round to zero.
  • CloudTrail: 90-day management-event history is free.

Pricing reference: AWS IAM. IAM itself is free; what you'll pay for is what the roles do in other services.

Done?

Once you've written, attached, simulated, and destroyed — you've done the IAM lifecycle. The next service is the one that holds the encryption keys IAM decides who can use.
